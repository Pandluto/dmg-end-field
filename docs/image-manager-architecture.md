# ImageManager 架构说明

## 目标

这份说明只回答一个问题：`ImageManager` 应该按什么层次组织，避免交互逻辑、文件逻辑、桌面桥接逻辑继续混在一起。

本说明只保留三层：

1. 交互层
2. 文件处理层
3. 额外通信层

不再额外发明更多概念层。

## 三层定义

### 1. 交互层

交互层负责“用户看到了什么、点了以后界面怎么变”。

它处理的内容只有：

- 当前浏览目录
- 当前高亮选中项
- 当前预览项
- 树展开状态
- 菜单、弹窗、toast、loading
- 用户触发操作后的界面状态更新

它不负责：

- 真实磁盘路径
- Electron IPC 细节
- dev / prod 目录差异
- 文件系统增删改查实现

建议承载文件：

- `src/components/ImageManagerPage.tsx`

### 2. 文件处理层

文件处理层是基石。

它负责所有资源的增删改查，以及与资源本身有关的规则。

它处理的内容：

- 扫描图片资源
- 创建目录
- 删除目录
- 重命名目录
- 导入图片
- 删除文件
- 重命名文件
- reveal 文件 / 目录
- 路径解析与路径校验
- 逻辑路径到真实物理路径的映射

它不负责：

- 页面高亮状态
- 当前预览状态
- 菜单是否打开
- 前端按钮是否禁用

这一层对上层暴露的应该是稳定的资源语义，而不是宿主实现细节。

建议统一资源语义：

- 文件路径：`assets/images/...`
- 目录路径：`images/...`

建议承载文件：

- `src/utils/imageFileService.ts`

如果桌面端实现需要保留在 Electron 主进程中，也应把主进程里的文件逻辑整理成这一层的实现，而不是让页面直接感知主进程细节。

### 3. 额外通信层

额外通信层负责把前端交互请求送到桌面端，再把结果带回来。

它本质上是“浏览器端和桌面端之间的桥”。

它处理的内容：

- 前端如何调用桌面能力
- Electron preload 暴露哪些接口
- IPC payload / result 的整理
- 浏览器端没有桌面能力时的只读回退

它不负责：

- 文件系统业务规则
- 页面状态恢复
- 磁盘路径推导

这一层应该只做“转发”和“边界整理”，不应该自己做业务判断。

建议承载文件：

- `src/utils/imageBridge.ts`

如果需要兼容现有 Electron 入口，则：

- `src/utils/imageBridge.ts` 作为前端统一入口
- `electron/preload.cjs` 继续提供桌面桥接能力
- `electron/main.cjs` 继续提供桌面实现

但页面只能依赖 `imageBridge.ts`，不能直接依赖 Electron 细节。

## 依赖方向

依赖方向必须单向：

- 交互层 -> 额外通信层
- 额外通信层 -> 文件处理层

交互层不能直接碰文件处理实现。

额外通信层不能反过来操作页面状态。

文件处理层不能感知页面状态。

## 当前问题本质

当前 `ImageManager` 的问题，不是单纯某个按钮坏了，而是三层职责串了。

具体表现为：

- 交互层在猜路径语义
- 通信层暴露了过多宿主实现细节
- 文件处理逻辑散落在页面和主进程之间

结果就是：

- 新建、删除、导入、reveal 看似是不同问题
- 实际都被同一套混乱边界影响

## 重构目标

后续重构时，只做下面这件事：

### 交互层只问

- 当前浏览什么
- 当前选中什么
- 当前预览什么
- 用户触发操作后界面怎么更新

### 文件处理层只问

- 资源在哪里
- 如何增删改查
- reveal 如何执行
- 路径是否合法

### 额外通信层只问

- 前端如何把请求送到桌面端
- 桌面端如何把结果回给前端

## 文件建议

最小落地建议就是三个核心文件：

- `src/components/ImageManagerPage.tsx`
- `src/utils/imageFileService.ts`
- `src/utils/imageBridge.ts`

如果 Electron 侧暂时不拆文件，也至少要让主进程中的 `ImageManager` 文件逻辑在职责上对应到“文件处理层”，不要继续让页面感知这些细节。

## 浏览器端与桌面端的区别

浏览器端和桌面端不建议拆成两套 `TSX` 页面。

建议保持：

- 一个 `ImageManagerPage.tsx`

差异放在额外通信层：

- 桌面端：提供可写能力
- 浏览器端：提供只读能力或明确的不支持结果

这样页面只根据能力决定：

- 哪些操作可用
- 点击后显示什么提示

而不是维护两套 UI。

## 后续 todo 的写法原则

后续 todo 不再按“页面上哪个按钮坏了”来写。

而是按三层拆解：

1. 交互层要保留哪些状态
2. 文件处理层要统一哪些增删改查接口
3. 额外通信层要收敛哪些桥接能力

这样才能避免继续出现“看起来像 UI bug，实际是底层路径或桥接问题”的误判。
