# 技术栈与技术选择

[← 返回项目入口](../README.md)

这不是一份把所有依赖都贴成徽章的清单。终末地伤害工作台最重要的技术选择，来自它需要在本地保存复杂方案、让 AI 真实操作，又不能让 AI 越过用户边界这三个问题。

## 核心技术选择

| 能力 | 采用 | 为什么它是重点 |
| --- | --- | --- |
| 桌面工作台 | Electron | 将配装、排轴、本地资料、SQLite 与 Agent 放在同一台机器上协作；不把有状态的工作流降级成一次性的网页表单。 |
| AI Runtime | OpenCode + DEF adapter | 项目内置、构建并启动 OpenCode 上游源码，再围绕终末地任务适配会话、技能和能力策略。 |
| Agent 执行边界 | Typed Tools | AI 只能经由角色、武器、装备、伤害和排轴等已定义操作做事，而不是获得任意 Shell、文件或目录权限。 |
| Agent 能力验证 | Harness | 用版本化工件和真实会话场景回放比较稳定版与候选版，检查修复、回归与用户状态安全。 |
| 排轴版本库 | SQLite Repository | 在本机保存文档、快照、AI 草稿、当前应用目标与审计事件，支持检查、恢复、分享和原子导入。 |

其中 Typed Tools 与 Harness 不应混为一谈：前者决定 **AI 现在允许做什么**，后者验证 **运行时改动以后仍能可靠地做这些事**。

## 支撑层

| 层次 | 采用 | 责任 |
| --- | --- | --- |
| 界面 | React 18 + TypeScript | 工作台页面、领域状态和计算结果呈现。它是界面基础，不是项目技术叙事的中心。 |
| 前端构建 | Vite | 提供本地开发服务器与 Web 构建。 |
| 桌面数据访问 | Electron 主进程 + 本地 REST/SSE bridge | 让渲染器在受控边界内访问本机能力；地址为 `127.0.0.1`，不是远程服务。 |
| 存储驱动 | Node `node:sqlite` / `DatabaseSync` | 本机 SQLite 文档库，启用 WAL、外键与同步配置。 |
| 构建发布 | electron-builder | 生成 Windows 便携版与 macOS DMG。 |

## 有意不采用的东西

- **没有云端排轴数据库或多人同步**：排轴的正式事实源是本机 AppData 下的 SQLite，而不是 Turso、远程 PostgreSQL 或浏览器缓存。
- **没有 Hono 服务端**：`127.0.0.1` 是 Electron 为桌面渲染层提供的本地桥接，不是独立部署的 Web 后端。
- **没有给 AI 任意本机权限**：模型侧的“会调用工具”不等于它可以自行读写任何文件或执行命令。

## 如何读这组技术

可以把项目看成四个互相制约的层次：Electron 提供本地工作台，SQLite 让方案成为可恢复的文档，OpenCode 与 Typed Tools 让 AI 在领域边界内协作，Harness 再验证这条协作链路不会因演进而退化。

具体的数据流与模块边界见[架构总览](./architecture/overview.md)；如何在本地启动与验证见[开发与启动](./guides/development.md)。
