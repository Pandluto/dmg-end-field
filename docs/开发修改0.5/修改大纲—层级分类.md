# 修改大纲—层级分类

[阶段目标]
- 本阶段目标是做前端内部层级分离，不是继续修单点数据 bug。
- 项目仍保持纯 TSX / React / sessionStorage 架构，不引入真实后端，不引入 Redux/Zustand。
- 目标是把“数据读写、业务规则、计算逻辑、事件契约”从 UI 组件中拆出去，形成类似前后端分离的维护边界。
- UI 组件只负责展示、交互入口和调用 application hook，不再直接维护缓存主从规则。

[当前层级现状]
1. UI 组件层
   - 目录：
     - `src/components/CanvasBoard`
     - `src/components/SidePanel`
     - `src/components/SelectionPanel`
   - 当前问题：
     - `DamageTab.tsx` 同时负责 UI、候选 Buff 加载、storage 写入、事件派发。
     - `SkillButton.tsx` 同时负责弹窗 UI、读取配置、读取 Buff、监听事件、伤害计算编排。
     - `CanvasBoard/index.tsx` 同时负责布局、恢复数据、迁移旧缓存、监听 Buff 事件、协调拖拽。
     - `OperatorConfigPanel.tsx` 同时负责配置 UI、状态、计算、快照、保存，文件过大。

2. 状态与应用逻辑层
   - 当前文件：
     - `src/context/AppContext.tsx`
     - `src/hooks/useTimelineData.ts`
     - `src/hooks/useSkillButtonBuffs.ts`
   - 当前问题：
     - `AppContext.skillButtons`、`timelineData`、`def.skill-button.v1`、`def.all-buff-list.v1` 并存。
     - hooks 同时承担 React 状态、业务规则、repository 调用职责。
     - 部分业务规则靠注释和约定维护，不是靠层级强制。

3. 数据持久化层
   - 当前文件：
     - `src/utils/storage.ts`
     - `src/constants/storage-keys.ts`
     - `src/utils/migrateStorage.ts`
   - 当前问题：
     - `storage.ts` 混合角色配置、计算缓存、旧 Buff map、新 skill-button 表、新 buff-list 表、通用 JSON 存取。
     - 组件和 hooks 可以直接 import storage 方法。
     - 没有明确 repository 边界。

4. 计算层
   - 当前文件：
     - `src/components/CanvasBoard/SkillButtonBuffCalculator.ts`
     - `src/utils/layout.ts`
     - `src/utils/nodeNumbering.ts`
     - `src/utils/equipmentParser.ts`
     - `src/utils/weaponFuzzySearch.ts`
   - 当前问题：
     - 部分纯计算已抽出，但 `SkillButton.tsx` 仍包含大量伤害计算编排。
     - `OperatorConfigPanel.tsx` 仍包含配置计算和快照生成逻辑。

5. 事件通信层
   - 当前散落事件：
     - `skillbutton-buff-added`
     - `skillbutton-buff-removed`
     - 全局 `mousemove / mouseup`
   - 当前问题：
     - 事件名是字符串散落。
     - payload 没有统一类型。
     - 事件监听方曾经参与错误写回主表，风险高。

[目标层级]
建议建立 `src/core`，作为独立的数据读写、业务规则、计算和事件层。

目标结构：
```text
src/core/
  domain/
  repositories/
  services/
  calculators/
  events/
```

1. `src/core/domain`
   - 存放稳定领域类型和领域约束。
   - 包含：
     - skill button 类型
     - buff 类型
     - timeline 类型
     - operator config 类型
     - damage calculation 输入输出类型
   - 要求：
     - 领域类型不依赖 React。
     - 不直接读写 sessionStorage。
     - 不包含 UI props。

2. `src/core/repositories`
   - 只负责 sessionStorage 数据读写。
   - 建议拆分：
     - `timelineRepository`
     - `skillButtonRepository`
     - `buffRepository`
     - `candidateBuffRepository`
     - `operatorConfigRepository`
   - 要求：
     - repository 是 storage key 的唯一直接使用者。
     - 组件层禁止直接调用 `safeSessionStorage`、`getStorageJson`、`setStorageJson`。
     - `def.all-buff-list.v1` 只能由 `buffRepository` 写。
     - `def.candidate-buff-list.v1` 只能由 `candidateBuffRepository` 写。

3. `src/core/services`
   - 负责业务规则和写入顺序。
   - 建议拆分：
     - `timelineService`
     - `skillButtonService`
     - `buffService`
     - `operatorConfigService`
   - 要求：
     - 主从关系规则只写在 service。
     - 清理顺序只写在 service。
     - UI 和 hooks 不再自己拼 storage 写入顺序。

4. `src/core/calculators`
   - 负责纯计算。
   - 建议迁移：
     - `SkillButtonBuffCalculator.ts`
     - 伤害计算编排
     - 面板快照计算
   - 要求：
     - calculator 不依赖 React。
     - calculator 不读写 storage。
     - calculator 输入输出必须是纯数据。

5. `src/core/events`
   - 负责事件名和 payload 契约。
   - 建议封装：
     - `emitSkillButtonBuffAdded`
     - `onSkillButtonBuffAdded`
     - `emitSkillButtonBuffRemoved`
     - `onSkillButtonBuffRemoved`
   - 要求：
     - 组件不得手写 `skillbutton-buff-added` 字符串。
     - 事件 payload 必须有类型。
     - 事件监听器不得写 storage 主表，只能触发 UI 刷新或调用 service。

[核心主从关系]
必须继续保持 0.4 阶段已经修正的数据主从关系。

1. `def.skill-button.v1`
   - button 主表。
   - 管 button 自身字段。
   - 管 button 的 `selectedBuff`。

2. `def.all-buff-list.v1`
   - 已选 Buff 实体主表。
   - 管完整 Buff 内容。
   - 只能保存已选 Buff 实体，不得保存候选列表。

3. `def.candidate-buff-list.v1`
   - 候选 Buff 列表缓存。
   - 只能服务 `DamageTab` 的候选展示。
   - 不参与已选 Buff 恢复。

4. `def.timeline.data.v1`
   - 排轴展示和恢复从结构。
   - 不再作为 Buff 关系主真相。
   - 不得从 `timelineData.buttons[].buffIds` 写回 `selectedBuff`。

[本阶段必须落实的边界]
1. 组件层边界
   - `DamageTab.tsx` 不直接写 `setStorageJson`。
   - `SkillButton.tsx` 不直接读 `getCharacterConfig` 和 Buff repository。
   - `CanvasBoard/index.tsx` 不直接参与 Buff 主表同步。
   - `OperatorConfigPanel.tsx` 的计算和保存逻辑逐步迁出。

2. Hook 层边界
   - hooks 只负责 React 状态、生命周期和调用 service。
   - hooks 不直接维护复杂业务顺序。
   - hooks 不直接处理跨表清理规则。

3. Service 层边界
   - `buffService` 统一负责：
     - 添加 Buff 到 button
     - 删除单个 Buff
     - 清空 button Buff
     - 删除 button 时清理 Buff
     - 引用检查
   - `timelineService` 统一负责：
     - 添加 button
     - 删除 button
     - 移动 button
     - 保存/恢复 timeline
   - service 不依赖 React。

4. Repository 层边界
   - repository 只负责读写，不负责业务判断。
   - repository 不做 UI 事件派发。
   - repository 不引入 React。

[必须下沉到 service 的规则]
1. Buff 添加规则
   - 先读取当前 button。
   - 先检查重复。
   - 再写 `buff-list`。
   - 再写 button 的 `selectedBuff`。
   - 返回实际 `buffId`。

2. Buff 删除规则
   - 先从 button 的 `selectedBuff` 解绑。
   - 再检查该 Buff 是否仍被其他 button 引用。
   - 无引用才删除 Buff 实体。

3. 清空 Buff 规则
   - 先保存旧 `selectedBuff`。
   - 先将当前 button 的 `selectedBuff` 写为空数组。
   - 再对旧 buffIds 做引用检查。
   - 无引用才删除 Buff 实体。
   - 这是 `clearBuffs()` 曾经出错的点，本阶段必须把该规则沉到 `buffService.clearButtonBuffs()`。

4. 删除 button 清理规则
   - 删除 button 前先读取旧 `selectedBuff`。
   - 删除 button。
   - 再对旧 buffIds 做引用检查。
   - 无引用才删除 Buff 实体。

5. Timeline 规则
   - `timeline.data` 只维护排轴展示和恢复。
   - 禁止从 `timelineData.buttons[].buffIds` 写回 `selectedBuff`。
   - 旧 `updateButtonBuffIds()` 只能 no-op 或删除。

[阶段执行顺序]
1. 新建 `src/core/repositories`
   - 先把 `storage.ts` 中的 skill-button、buff、candidate、timeline 读写封装成 repository。
   - 暂时允许 repository 内部复用旧 `storage.ts`。
   - 不先删除旧 storage 函数。

2. 新建 `src/core/services`
   - 先抽 `buffService`。
   - 再抽 `timelineService`。
   - 把添加、删除、清空、删除 button 清理、引用检查迁入 service。

3. 改 hooks 调用 service
   - `useSkillButtonBuffs.ts` 改为调用 `buffService`。
   - `useTimelineData.ts` 改为调用 `timelineService`。
   - hooks 只保留 React 状态同步。

4. 改组件调用 hook / service facade
   - `DamageTab.tsx` 不直接写 candidate storage。
   - `SkillButton.tsx` 不直接读 storage。
   - `CanvasBoard/index.tsx` 只处理布局和 UI 协调。

5. 抽计算层
   - 把 `SkillButton.tsx` 中的伤害计算编排迁到 calculator/service。
   - 后续再拆 `OperatorConfigPanel.tsx`。

[不要做]
- 不要引入新状态库。
- 不要重写 UI。
- 不要改变 sessionStorage key。
- 不要把 `timelineData.buttons[].buffIds` 恢复为主真相。
- 不要把候选 Buff 写回 `ALL_BUFF_LIST`。
- 不要在普通函数里调用 Hook。
- 不要在组件里新增直接 storage 写入。

[验收标准 AC]
- AC1: 组件层不再直接调用 `safeSessionStorage`。
- AC2: `DamageTab.tsx` 不再直接调用 `setStorageJson`。
- AC3: `SkillButton.tsx` 不再直接读取角色配置 storage。
- AC4: `useSkillButtonBuffs.ts` 的添加、删除、清空逻辑下沉到 `buffService`。
- AC5: `useTimelineData.ts` 的 button 增删改移动逻辑下沉到 `timelineService`。
- AC6: `clearBuffs` 的“先解绑、再查引用、再删实体”规则只在 service 中维护。
- AC7: `ALL_BUFF_LIST` 只通过 buff repository 写入。
- AC8: `CANDIDATE_BUFF_LIST` 只通过 candidate repository 写入。
- AC9: `npm run build` 通过。

[回归检查项]
- 添加多个 Buff 后，关闭重开仍显示全部 Buff。
- 刷新候选 Buff 后，已选 Buff 实体表不被覆盖。
- 删除单个 Buff 后，引用和实体清理正确。
- 清空 Buff 后，无引用实体被删除。
- 删除整个按钮后，无引用 Buff 被删除。
- 跨谱线移动后，button id 和 selectedBuff 不丢。
- 刷新页面后，timeline、button、buff 均恢复。
