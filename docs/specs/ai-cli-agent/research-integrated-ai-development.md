# 综合 AI 开发报告

## 1. 目标

### AI 填表
建立“分段抽取 + 字段白名单 + Schema 约束 + 清洗复核”链路，通过 `sanitize`、`validate`、`convert` 流程过滤非法字段、占位符和脏数据，提升复杂描述自动落表的可用性。

### AI 校验
设计基于本地完整快照的单 Agent 提醒系统，主链路负责结构化落表，Agent 结合原始输入、draft、validator 输出和历史反馈识别缺项、冲突项与未映射术语，并生成可解释提醒。

## 2. 当前项目基础

- 主链路已具备：文本分段、AI 抽取、Schema 白名单约束、结果清洗、结构校验、固定 Draft 落表。
- 数据模型当前是固定 Schema，不是开放式自定义字段平台。
- 本地状态可从 `localStorage`、当前 draft、原始输入、页面缓存中恢复，具备单次 Agent 分析所需上下文。

## 3. 开发分层

### 3.1 AI 填表主链路

#### `Section Splitter`
- 按标题和来源切段。
- 目标：降低单次抽取上下文噪声。

#### `Extractor`
- 模型只做受限抽取。
- 输出必须落在固定字段白名单内。

#### `Schema Constraint`
- 用 JSON Schema 约束字段形状、字段枚举和必填项。
- 目标：阻断自由发挥型脏输出。

#### `Sanitizer`
- 过滤非法 effect。
- 删除占位符、零值脏项、无效 extraHit。
- 必要时修正可归一问题。

#### `Validator`
- 做结构正确性和业务白名单校验。
- 这是主链路唯一真相源。

#### `Converter`
- 将 AI 草稿转换成固定 `BuffDraft`。
- 供页面继续编辑、保存、导出。

### 3.2 AI 校验提醒链路

#### `Context Loader`
- 收集原始输入、当前 draft、validator 结果、本地别名、忽略记录。
- 生成一次分析快照。

#### `Reminder Agent`
- 不改 draft，不推翻 validator。
- 只做提醒分析。
- 识别三类问题：
  - 缺项
  - 冲突项
  - 未映射术语

#### `Evidence Builder`
- 为每条提醒生成证据链。
- 证据来源限定为原文、draft、validator 输出、本地反馈。

#### `Reminder Decision`
- 将分析结果压成固定提醒类型。
- 第一版不做复杂评分，只输出：
  - `possible_missing`
  - `possible_conflict`
  - `unmapped_term`
  - `check_manually`

#### `Feedback Memory`
- 记录用户忽略、确认别名、少量特例短语。
- 目标不是训练模型，而是增强下一次本地分析上下文。

## 4. 技术选型

### AI 填表
- 模型：结构化输出能力稳定的通用 LLM。
- 输出协议：JSON Schema constrained output。
- 原因：当前项目是固定字段落表，优先要可控，不要开放推理。

### AI 校验
- Agent 形态：按钮唤起的单 Agent。
- 编排方式：本地快照 + 单次分析，不做多 Agent，不做自治循环。
- 原因：当前项目更需要可解释提醒，不需要第二套主判定系统。

### 本地存储
- 继续使用 `localStorage` 承载 prompt、原始输入、draft、提醒忽略项、别名映射。
- 原因：当前是本地编辑器形态，本地快照即可满足提醒分析。

## 5. 项目落脚点

### 第一阶段：做强 AI 填表主链路
- 保持固定 Schema。
- 强化 `sanitize + validate + convert` 的稳定性。
- 目标：先把“能落准表”做稳。

### 第二阶段：补轻量 AI 校验
- 在 `validator` 后挂只读提醒层。
- 不改现有真相源。
- 目标：先给出高价值提醒，不做自动纠错。

### 第三阶段：沉淀本地反馈
- 增加别名映射、忽略记录、少量特例记忆。
- 目标：降低重复误报，提高下一次提醒质量。

## 6. 结论

- AI 填表与 AI 校验应拆成两条链路。
- 填表链路的核心是“受限抽取 + 强约束落表”。
- 校验链路的核心是“本地快照 + 单 Agent 提醒 + 可解释证据”。
- 当前项目不适合直接上多层自治 Agent，更适合“结构化主链路 + 只读提醒 Agent”的组合。
