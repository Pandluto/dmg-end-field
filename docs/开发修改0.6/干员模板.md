# 干员模板

## 目标

- 本模板用于“编辑层 / 角色创建器”导出的干员定义。
- 本模板不是原始角色资料，不追求保留 `9 / M3` 等成长档位。
- 本模板是软件层最终消费的角色真相。

## 设计原则

### 1. 贴住现有项目字段

优先复用当前项目已有字段，不重新发明一套完全平行的命名系统。

当前项目已有核心字段来源：

- `src/types/index.ts`
- `public/data/characters/*/*.json`

因此角色层优先保留：

- `id`
- `name`
- `avatarUrl`
- `rarity`
- `profession`
- `weapon`
- `element`
- `mainStat`
- `subStat`
- `level`
- `attributes`
- `skills`

### 2. 不再使用 9 / M3 多档位

- 编辑层导出的是最终结算模板，不是成长曲线。
- 不再保留：
  - `damage.9 / damage.M3`
  - `multipliers.9 / multipliers.M3`
  - `skillLevelMap`
  - `SkillLevelMode`

最终值直接写死在模板里。

### 3. A / B / E / Q 只做修饰，不做主真相

旧模型的问题是：

- 一个 skill 默认只有一个 `skillType`
- 整个 skill 默认只有一个伤害属性

新模板中：

- `buttonType` 只用于按钮分类 / UI 展示 / 技能沙盒入口归类
- 真正参与结算的是每个 `hit` 自己的元信息

### 4. skill 是 hit 的集合

- 一个 skill 可以是普通技能
- 也可以是混合技能
- 也可以是第二终结技、第二战技、自定义技能

每个 skill 下面：

- `hitMeta.hit1 / hit2 / hit3 ...` 作为唯一 hit 真相

### 5. 每个 hit 自己决定结算属性

每个 `hit` 自己拥有：

- `displayName`
- `element`
- `skillType`

因此允许：

- 同一个 skill 内
- `hit1` 吃物理 + 战技加成
- `hit2` 吃寒冷 + 终结技加成

## 字段说明

### 角色层

#### `id`

- 干员唯一 ID
- 主键
- 不允许依赖 `name`

#### `name`

- 干员显示名
- 可自定义

#### `avatarUrl`

- 干员头像资源路径
- 编辑层直接提供
- 软件层优先使用该字段，不再依赖角色名反推头像路径

#### `level`

- 当前模板角色等级
- 可自定义
- 不再把等级编码进 `attributes` 的 key 名里

#### `attributes`

- 当前等级下最终生效的基础值
- 不是成长曲线
- 不再使用 `attributes.level90 / level80 ...` 这种原始资料结构

字段继续沿用：

- `strength`
- `agility`
- `intelligence`
- `will`
- `atk`
- `hp`

### 技能层

#### `skills`

- 改为可扩展 map
- 不再固定死 `normalAttack / skill / chainSkill / ultimate`
- key 可以是：
  - `normalAttack`
  - `skill`
  - `ultimate2`
  - `mixBurst`
  - `customSkill1`

#### `displayName`

- 技能显示名
- 可完全自定义
- 编辑层不再强制沿用原项目 skill 层的 `name`

#### `iconUrl`

- 当前 skill 的图标资源路径
- 软件层优先使用该字段，不再依赖 `buttonType + 角色名` 反推技能图标

#### `buffSlots`

- 技能层预留 Buff 槽位
- 当前先占位，不定义具体行为
- 第一版最小写法：

```json
"buffSlots": []
```

#### `buttonType`

- 只做按钮类型修饰
- 当前值仍建议使用：
- `A`
- `B`
- `E`
- `Q`
- 作用：
  - 按钮显示
  - 技能沙盒归类
  - 默认图标归类
- 不作为最终伤害结算主真相

这意味着：

- `buttonType = Q` 的 skill
- 某个 `hit` 仍然可以写 `skillType = B`

#### `hitCount`

- 继续保留
- 表示当前 skill 有几个 `hit`
- 应与 `hitMeta` 中实际定义的 hit 数一致

### hit 层

#### `hitMeta`

- 新增
- 这是新模型的核心
- key 本身就是 hit 真相
- 倍率直接写在 `hitMeta.hitN.multiplier`

示例：

```json
"hitMeta": {
  "hit1": {
    "multiplier": 1.2,
    "displayName": "物理斩击",
    "element": "physical",
    "skillType": "B"
  }
}
```

#### `hitMeta.hitN.element`

#### `hitMeta.hitN.displayName`

- 当前 hit 的显示名
- 可自定义

#### `hitMeta.hitN.element`

- 当前 hit 实际吃的元素乘区
- 如：
  - `physical`
  - `fire`
  - `ice`
  - `electric`
  - `nature`

#### `hitMeta.hitN.skillType`

- 当前 hit 实际吃的技能乘区
- 当前仍建议沿用：
  - `A`
  - `B`
  - `E`
  - `Q`

这意味着：

- 一个 `buttonType = Q` 的 skill
- 某个 hit 可以写 `skillType = B`

#### `hitMeta.hitN.multiplier`

- 当前 hit 的最终倍率真相
- 不再额外保存在 skill 层 `damage.hitN`
- 例如：
  - `0.41`
  - `2.4`
  - `5.6`

## 汤汤类型复杂技能的表达方式

汤汤这种复杂机制，不再通过：

- 把水龙卷硬塞成 `ultimate.hit3`
- 或让整套 skill 只拥有一个固定 `skillType`

而应该改成：

- 每个技能都是独立 skill
- 每个 hit 自己声明：
  - `element`
  - `skillType`
  - `displayName`
  - `tags`

例如：

- `Q`：终结技正常版
- `Q2`：提前下落版
- `B`：战技本体
- `B2`：战技 + 特殊水龙卷版
- `Mix1`：一个技能内混合多个技能乘区和元素乘区

每个 skill 都是最终结算模板，不再依赖运行时动态拼装。

## 不再保留的旧思路

以下内容不进入编辑层模板：

- `9 / M3` 多档位
- `skillLevelMap`
- `extensions`
- 通过动态追加 hit 来临时拼技能
- 让整套 skill 只吃单一 `skillType`

## 软件层对接原则

软件层未来接编辑层时，主入口应改为：

1. 按钮绑定 `skillKey`
2. 读取该 `skill.hitMeta`
3. 逐个 `hitN` 结算：
   - 元素乘区
   - 技能乘区

不再使用旧假设：

- “当前按钮是 Q，所以整套技能都吃 ultimateDmgBonus”

而是：

- “当前 hit 的 `hitMeta.skillType = Q`，所以这一段吃 ultimateDmgBonus”

## 当前最小结论

编辑层第一版先定成：

- 保留现有角色字段风格
- 角色层只保留结算和基础分类必需字段
- `hitMeta.hitN` 作为唯一 hit 真相
- `skills` 改成可扩展 map
- `buttonType` 降级为修饰字段
- `level + attributes` 替代 `attributes.level90`
- 去掉与当前伤害计算无关的 `imbalanceValue`

这是目前最贴着现有项目、同时能表达复杂机制的最小主结构。
