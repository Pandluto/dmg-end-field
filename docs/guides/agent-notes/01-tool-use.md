# Tool Use：让模型碰到外部世界

模型不能直接读取当前配装，但程序可以。我们把这项能力包装成工具，告诉模型它叫什么、什么时候用、参数怎样填写；真正读取产品数据的函数仍留在程序里。

一个最小工具通常有四部分：

```text
name         工具叫什么
description  什么情况下应该用
schema       参数长什么样
handler      程序实际执行什么
```

例如，读取当前配装的工具可以这样描述：

```json
{
  "name": "read_current_loadout",
  "description": "读取当前时间线已经应用的配装",
  "parameters": {
    "type": "object",
    "properties": {
      "character": { "type": "string" }
    },
    "required": ["character"]
  }
}
```

模型看到名称、描述和参数格式，可能返回 `{"character":"A"}`。Runtime 先验证参数，再调用 handler；读取结果不会停在程序内部，而是作为 Tool Result 写回 Session，让模型据此决定下一步。

## 为什么工具要有类型

如果工具只接收一段随意文本，执行方还得重新猜模型的意思。**Typed Tool** 用 Schema 把输入收窄成明确结构：角色名放在哪里，哪些字段必填，数字和字符串能否混用，都可以在执行前检查。

类型不只是为了少报错。稳定的调用对象还能被权限规则识别，被日志记录，也能在失败后重新定位。模型说过什么是一回事，程序准备执行什么则应该有一份清楚的结构。

不过，Schema 只回答“参数是否合法”，不回答“这件事该不该做”。角色名填写正确，不代表模型可以直接替换武器；调用结构完整，也不等于用户已经批准。

工具注册同样不代表每轮都向模型开放。Runtime 还会根据当前 Agent、Session、Skill 和权限，筛选本轮真正可用的工具。模型只能调用它此刻看得到的能力。

到这里，模型终于能准确提出一次操作。接下来的问题是：读取可以直接完成，修改是否也能直接完成？这层边界记在 [Permission 与 Hook](./02-permission-and-hooks.md) 里。
