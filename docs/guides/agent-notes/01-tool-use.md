# Tool Use：让模型碰到外部世界

模型不能直接执行函数。所谓 Tool Use，是程序先把一组工具的说明交给模型，模型在合适的时候生成一份结构化调用请求，再由程序完成真正的执行。

一个最小工具通常有四样东西：

```text
name         工具叫什么
description  什么情况下应该用
schema       参数长什么样
handler      程序实际执行什么
```

比如“读取文件”可以描述为：

```json
{
  "name": "read_file",
  "description": "读取指定路径的文本文件",
  "parameters": {
    "path": "string"
  }
}
```

模型看到的是前面三项，看不到 handler 里的实现。它可能返回 `{"path":"docs/README.md"}`，程序验证参数后调用真正的读取函数，再把文件内容作为 Tool Result 写回 Session。

## 为什么强调 Typed Tool

如果工具只接受一段随意文本，执行方还要重新猜模型的意思。Typed Tool 把参数限制成明确的结构：哪些字段必填、哪些只能取固定值、数字范围是多少。这样既减少误解，也让权限判断、日志记录和错误处理有稳定对象。

Schema 解决的是“参数是否合法”，并不保证“这件事应该做”。路径格式正确，不代表模型就有权读取那个路径；修改参数完整，也不代表可以绕过用户审批。权限是下一层问题。

## 注册和调用是两件事

工具注册只是把能力放进 Registry。真正发给模型前，程序还会根据当前 Agent、Session、Skill 和权限做一次筛选。模型只能调用本轮实际暴露给它的工具。

工具执行后，结果也不是直接展示给用户就结束。Agent Loop 会把结果重新放进上下文，让模型决定下一步：继续查资料、修正参数，还是组织最终回答。这一步叫结果回注。没有它，工具只是一个外部按钮，还不能形成连续工作。

Tool Use 给了 Agent 一双手。手能做的事情越多，风险也越大。下一则笔记谈 [Permission 与 Hook](./02-permission-and-hooks.md)：谁来决定这双手可以伸到哪里。
