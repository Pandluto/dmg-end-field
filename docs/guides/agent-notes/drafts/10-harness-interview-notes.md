## 第一轮：PSM（问题求解方法）为什么出现在后半篇

### 用户原话

后半篇什么意思呢？就是我先回答你第一个问题。这是一篇开发手记，终将要把概念型的问题落到工程上的问题。PSM 这个问题解决模型。是知识工程的一个特点。但是它跟我们开发垂直领域的智能体有很紧密的联系性。读者不会开发智能体。但读者可能都是做题家。虽然都是概念，但是 PSM 相当于直接讲 harness 更好让读者理解。PSM 讲的是什么呢？一个问题、一个 task 怎么去解决，要怎么做？而智能体它实际是怎么运行的呢？分析和理解读者发出来的对话内容。调用需要的 tools。得到结果，再进行加工，退出循环得到答案。而 harness 的体系呢，作为这一套的润滑者和管理者。而问题解决模型呢可能是个论证证明题。我们要用调用哪些证明方法去加工？怎么一个链路？最后得到什么结论？

### 暂时整理

- 这是一篇**开发手记**。后半篇必须把前面的概念讨论落到工程问题，而不是继续增加概念。
- 读者未必开发过智能体，但熟悉“解题”。因此，PSM 是理解 Harness 的**认知桥梁**。
- PSM 先回答：一个问题或 Task（任务）要怎样解决，需要采用哪些方法，经过怎样的加工链路，最后得到什么结论。
- 智能体的基本运行过程是：理解用户对话，调用所需 Tools（工具），取得结果，继续加工，直到退出循环并给出答案。
- Harness 不是这套解题方法本身，而是围绕这套运行过程发挥**润滑和管理作用**。
- “论证证明题”是当前最重要的通俗类比：需要选择证明方法，组织推导链路，并得到最终结论。

### 当前仍待追问

- PSM 这座认知桥，具体怎样继续落到 Harness 的工程结构上。
- “润滑者”和“管理者”在实际运行中分别管理什么。
- 这条主线与后文的 Runtime（运行时）、业务拆分和版本管理应当怎样衔接。

## 第二轮：后半篇的主命题与术语主次

### 用户原话

不使用 harness、PSM、runtime 这些词。很难一次性讲清楚后半段的内容。本篇 Harness 随记的最大主题，其实是这个：它将领域知识、问题求解方法、上下文、能力边界、执行状态与完成验证组织为可版本化、由运行时强制的整体，同时保留模型在边界内的自主推理能力。

### 用户提供的参考

对。绕了一圈，结论就是：**Harness 作为主名称反而最合适。**

因为它足够宽，能够同时容纳：

- 提示词与程序性知识；
- 真实上下文；
- 解题方法；
- Tool 能力边界；
- 事务状态；
- 阶段推进；
- 版本管理；
- 审批与结果验证。

其他术语只能解释它的一面：

- PSM 解释“解题方法”；
- Workflow 解释“执行路径”；
- Policy/Guardrail 解释“强制约束”；
- Runtime 解释“运行载体”；
- Skill 解释“教学内容”。

但 Harness 是这些东西围绕 Agent 形成的整体。

因此你最开始的直觉是对的。只需要在文档开头给它一个项目内的严格定义：

> 本项目中的 Harness，是围绕一类业务问题建立的完整 Agent 求解环境。它将领域知识、问题求解方法、上下文、能力边界、执行状态与完成验证组织为可版本化、由运行时强制的整体，同时保留模型在边界内的自主推理能力。

术语可以最终定成：

- 整体：`Harness System` / `Harness Runtime`
- 单元：`Business Harness`
- 架构：`五业务原子 Harness 架构`
- 原子性：按完整业务责任、写入边界和版本生命周期划分
- PSM、Workflow、Policy：都作为解释 Harness 某一侧面的理论参照

所以 Spec9-2 继续叫“原子化 Harness”没有问题。重点不是再寻找一个替代词，而是明确：

> **Harness 是主概念；PSM、Workflow、Skill 和 Guardrail 都只是它内部的不同组成与观察角度。**

### 暂时整理

- 后半篇的**唯一主概念是 Harness**。PSM 不是并列主线，而是帮助非智能体开发者理解“解题方法”这一侧面的认知桥。
- PSM、Workflow（工作流）、Policy / Guardrail（策略 / 护栏）、Runtime、Skill 分别解释 Harness 的一个侧面，不能取代 Harness 作为整体名称。
- 本项目对 Harness 的严格定义已经明确：它是围绕一类业务问题建立的完整 Agent 求解环境。
- 这个整体同时组织六类内容：**领域知识、问题求解方法、上下文、能力边界、执行状态、完成验证**。
- 工程上的两个关键性质是：**可以版本化**，并且**由 Runtime 强制边界**。
- 模型没有被 Workflow 完全取代；它仍在被强制的边界内保留自主推理能力。
- 后续叙事不应继续寻找 Harness 的替代词，而应说明这些侧面怎样被 Harness 组织成一个可以运行的整体。

### 当前仍待追问

- “由 Runtime 强制”的准确边界：Runtime 强制整个 Harness，还是只强制其中可观察、可验证的部分。
- `Harness System`、`Harness Runtime` 与 `Business Harness` 三者在工程实现中的包含和执行关系。
- 六类内容分别以什么形式进入一个可版本化的 Business Harness。
- “五业务原子 Harness 架构”为什么正好按完整业务责任、写入边界和版本生命周期划分。

### 学者确认

- 主从关系已经确定：Harness 是主概念，其余术语解释它的不同侧面。
- 后半篇的核心不是“Agent 怎样调用几个 Tools”，而是怎样把一项领域问题所需的知识、方法、上下文、能力边界、状态和完成验证组成可运行、可迭代的整体能力。
- 工程分层初步确定：Harness System 是整体基础设施，Harness Runtime 负责运行，Business Harness 负责一类完整业务。
- 原子边界不按文件、字段或 Tool 划分，而按完整业务责任、写入边界和版本生命周期划分。
- 下一处关键断点是：“**可版本化、由运行时强制的整体**”在项目中究竟具有什么工程含义。

### 下一轮问题：整体怎样成为可运行的工程对象

1. 以配装 Business Harness 为例，一个可以发布的具体版本由哪些项目对象构成？请尽量对应真实的文件、配置、注册项、提示内容、状态结构或程序入口。
2. 一次配装任务固定到某个 Harness 版本时，究竟固定了什么？攻略或公式更新后，已开始的任务与新任务分别会读取什么？
3. 沿一次真实配装请求，Harness Runtime 在哪些位置介入？哪些环节只是把材料交给模型，哪些会由程序直接拒绝或中止？
4. 同一个配装目标下，能否给出两条都合法但推理过程不同的执行路径，再给出一个 Runtime 必须拦截的动作？
5. 配装任务暂停后再恢复，除了对话记录，Runtime 最少必须保存哪些业务状态？为什么不能只从聊天内容恢复？
6. 配装任务的完成验证具体检查什么权威状态与结果？Tool 返回成功但验证不通过时，任务处于什么状态？
7. Business Harness 声明业务规则后，通用 Runtime 怎样强制规则，而不把五项业务逻辑重新写死？可以用“应用配装前必须确认”说明声明、读取与拦截分别发生在哪里。
8. 哪些变化算 Business Harness 的新版本，哪些只算 Harness System 或 Runtime 的升级？请各举一个具体变化。

## 第三轮：PSM 与 Harness 不能等同

### 用户原话

能否拿一道普通证明题说明？这个问题很好啊。我提出 PSM 它并不等于 harness。PSM 本身不是一个智能体，它没有“智能”这概念，而我们的 harness 是为智能体而生的。你所告诉我的，哪些是题目本身，哪些可复用临场判断，这些东西本质上是需要 harness 来做的。PSM 这个东西，就是用更通俗的方法把围绕 harness 的东西展示出来，也就是它需要展示的组织和管理方面的东西。你就说这句话。

### 暂时整理

- **PSM 不等于 Harness。**
- PSM 本身不是智能体，也不包含“智能”或自主判断这一前提。
- Harness 是围绕智能体建立的；它要面对题目、可复用方法和模型临场判断之间的组织关系。
- 引入 PSM 的作用是通俗解释，不是建立第二套与 Harness 并列的工程体系。
- 证明题不需要在正文中展开成复杂案例。文章只需用一句话说明：PSM 借“怎样解一道题”这个熟悉视角，帮助读者看见 Harness 要为智能体组织和管理什么。

### 可直接进入正文的核心句

> **PSM 不是 Harness，也没有智能；它只是借“怎样解一道题”这个熟悉视角，让读者看见 Harness 需要为智能体组织和管理题目、方法与临场判断。**

## 第四轮：拒绝把 Harness 问成一张固定映射表

### 被用户否定的问题

> 把证明题映射到“给当前角色配装备”时，题目、PSM、领域知识、Tools 和模型判断分别对应什么？

### 用户原话

关于第 2 个问题，映射到配装题目，PSM 领域知识 tools and models 判断分别是什么？这个问题呢，就非常的像面试官想拷打你。当我们聊这个 harness 可迭代和自训练的问题时，你非要问一个正则化的匹配。这是个很无聊的问题。当你把这一切都写死，你还有什么意义？到底是你在管这些问题，还是 harness。

### 暂时整理

- 这个问题错误地预设了：题目、PSM、领域知识、Tools 和模型判断之间应该存在一张固定、标准化的对应表。
- 这种问法会把 Harness 重新写成静态 Workflow，违背模型在边界内自主推理的前提。
- 如果作者预先把所有匹配关系写死，那么真正管理问题的是作者，不是 Harness。
- 后半篇要讨论的是 Harness 的**可迭代与自训练**：这些要素怎样在运行中被组织、被观察，并继续调整，而不是怎样被人工永久配对。
- 后续访谈应追问运行中的组织、反馈和演进机制，不再要求为每个概念寻找唯一对应物。

### 必须保留的反问

> **当你把这一切都写死，还有什么意义？到底是你在管这些问题，还是 Harness 在管？**

### 学者修正

此前的问题错误地预设了任务、方法、知识、工具和判断之间存在稳定的一一对应关系，只要由作者完整列出，Harness 就算讲清楚了。

这实际把“理解 Harness”偷换成了“由写作者预先解完问题”。一旦正确路线、材料和步骤都能被人工穷举，剩余部分更接近 Workflow，而不是允许模型在运行中判断、接受反馈并继续演进的 Harness。

“可版本化”也不应只被理解为冻结静态结构。版本更重要的意义可能是：让每一次组织方式的变化能够被观察、评价、继承、比较和回退。

### 下一轮问题：动态闭环

1. Harness“自训练”的最小闭环怎样发生？从一次真实业务运行到下一次运行因此改变，中间留下什么、怎样被判断，又怎样转化成新行为？
2. 运行中出现预先没有写好的情况时，Harness 和模型怎样共同组织下一步？模型发现证据不足以后，这个发现怎样进入系统，Harness 又怎样继续提供条件、接住行动并保留过程？
3. 一次运行中的什么反馈值得进入后续迭代？怎样判断它不是只适用于这一次的偶然个案？
4. 一次运行所得的经验什么时候可以直接影响后续任务，什么时候必须先形成新版本？运行中调整、跨运行学习与正式版本发布是什么关系？

后续研究不再替 Harness 列完“正确解法”，而是追问开放的解题过程如何保持：

> **可组织、可观察、可反馈、可演进，同时不失去边界。**

## 第五轮：Harness 的“强参考”性质

### 用户原话

Tools 是强制性的东西。Skills 是一个非强制性的目录。Workflow 是一个强制性串联 tools 的东西。而我们要实现的目的，是让 harness 达到一个非强制性但强参考的效果。虽然 Harness 不能跟他们一概而论，它是把这三者甚至更多串联起来的，对吧？但它最终的效果跟他们可以对比，它是强参考的。这个之前我在跟你讲 harness 跟 tools、skills 和 workflow 的关系，以及和 runtime 的关系中提到过，我不知道你还记得吗？

### 暂时整理

- Tool（工具）提供由程序执行和约束的能力，具有强制性。
- Skill（技能包）是一份非强制的方法目录；被加载不等于模型一定照做。
- Workflow（工作流）把 Tools 按确定顺序串联，具有强制性。
- Harness 不能与三者作为同一层级的部件直接并列，因为它还会组织这些部件及更多运行条件。
- 但可以比较它们最终作用于 Agent 的方式：Harness 对模型的问题求解应当达到**非强制、但强参考**的效果。
- “强参考”是此前关系辨析中最重要、但在当前正文中被弱化的比较轴。

### 与 Runtime 的关键区分

- Harness 为模型提供强参考，不强制模型采用唯一思考路径。
- Runtime（运行时）强制的是可观察的能力边界、状态条件和执行后果。
- 因此，“Harness 是由 Runtime 强制的整体”不能被写成“Runtime 强制模型遵循 Harness 中的全部方法”。
- 更准确的方向是：**Runtime 执行 Harness 中可强制的边界；Harness 中的问题求解方法则作为模型的强参考。**

### 学者确认

“非强制但强参考”与“由 Runtime 强制”并不冲突。关键在于，同一个 Harness 中的内容对模型具有不同效力：

- 关于“怎样解决这类问题”，Harness 提供非强制但强参考的方法。若锁定唯一思考与行动路径，就会退化成 Workflow。
- 关于“哪些动作可以真实发生、当前状态是否允许、结果能不能算完成”，Runtime 强制检查可观察的行为及其后果。

这条区分可以收束为：

> **方法不强制，边界和后果强制。**

它也解释了 Harness 为什么具有迭代和自训练空间：模型可以在强参考下产生不同的合法解题过程，系统才能观察结果、比较轨迹并继续改进；如果路线已经被 Workflow 全部写死，就没有这种空间。

### 下一处最小边界

> 如果模型没有采用 Business Harness 推荐的解题方法，但没有越过任何能力和状态边界，而且最终结果通过了完成验证，这次任务是否仍应被视为成功？

如果答案是“是”，那么“非强制但强参考”的工程含义就是：Harness 保证模型能够看见并依靠一套强方法，但 Runtime 不以“是否照步骤执行”为成功条件，只以越界与真实结果作硬判断。
