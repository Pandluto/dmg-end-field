# 篇章 11：Harness（运行支架）

前面已经给 Agent（智能体）接上了 Typed Tools（类型化工具）。

它能读取当前队伍和配装，能修改装备、编辑排轴、添加 BUFF（增益效果），也能调用真实公式完成计算。到这里，Agent 算是有手了。

然后用户说：

> 帮我给当前角色搭一套适合当前队伍的装备。

这句话没有要求调用哪个 Tool（工具），也没有给出固定步骤。

它只给了一个结果：一套真正适合当前队伍的配装。

接着才轮到这些问题：

- “当前角色”到底是谁，在队伍里承担什么作用？
- 这个角色现在穿什么，队伍里还缺什么？
- 哪些攻略适用于当前队伍？
- 哪些装备真的存在，数值是什么？
- 用户只是想听建议，还是准备修改？
- 等用户确认时，原来的方案有没有被别人改过？
- Tool 说成功以后，真实页面到底变没变？

这些问题不能从 Tool 列表里直接得到答案。它们共同决定的是：**这个 Agent 怎样完成一项真实业务。**

当然也可以写一个 `auto_loadout`，把整套配装逻辑塞进代码。这样很快，但 Agent 学会的只是调用配装程序，并没有学会处理配装问题。

就像给一个只会读稿的人接上装备查询和修改 Tools。他有手了，仍然不会做攻略。因为 Prompt（提示词）还在教他读稿，Skill（技能包）里是读稿经验，Context（上下文）里也只有稿件。

那把 Prompt 里的“读稿”改成“做攻略”呢？

也不够。旧方法可能还留在 Host Prompt（宿主提示词）、Skill 和 Tool Description（工具描述）里；权限、审批、任务进度和真实产品状态，也不会跟着一句 Prompt 一起变化。

问题于是从“改哪一句 Prompt”，变成了：

> 模型周围这整套工作条件，究竟由谁组织？

这就是这篇手记里的 Harness。

## 什么是 Harness（运行支架）？

先给一个不严谨、但好理解的说法：

> **Harness 是模型的工作台。**

模型负责判断下一步。Harness 把这一刻需要的材料和能力摆到桌上，接住模型提出的动作，再处理动作产生的结果。

  <div class="accent"><small>准备 · 约束 · 反馈</small><strong>Harness</strong></div>

再说准确一点：

> **Harness 是围绕模型建立的运行机制。**

它决定模型在什么信息和能力条件下工作，模型输出怎样被系统接收、执行、拒绝、记录和反馈，一件尚未完成的工作又怎样继续下去。

所以 Harness 不是单独一份 Prompt，也不是一组 Tools。

同一个模型接上不同 Harness，会表现得像不同的 Agent。不是模型突然变聪明了，而是它醒来以后，看见的事实、拿到的能力、受到的限制和面对的完成标准都变了。

## 广义 Harness（运行支架）会指什么？

Harness 没有一条被所有框架共同采用的边界。

最窄的说法，可能只指 System Prompt（系统提示词）和 Tool Loop（工具调用循环）；更宽的说法，会把 Context、Memory（记忆）、Skills、状态、权限、沙箱、审批和结果检查都算进去。

因此，看到别人讨论 Harness，第一件事不是争定义，而是先问：

> 他把边界画到了哪里？

这篇采用宽一点的说法。广义 Harness 是模型与真实环境之间的**运行支架**：它让模型能够持续取得信息、提出行动、接收反馈并延续状态。[AI Harness Engineering](https://arxiv.org/abs/2605.13357)（《人工智能运行支架工程》）也把 Agent 能力放在“模型—Harness—环境”这个整体中讨论。

在这个项目里，还可以把定义再收紧一点：

> **本项目中的 Harness，是围绕一类业务问题建立的完整 Agent 求解环境。它将领域知识、问题求解方法、上下文、能力边界、执行状态与完成验证组织为可版本化、由运行时强制的整体，同时保留模型在边界内的自主推理能力。**

这句话有一个容易误读的地方。

“由运行时强制”不是强制模型照着某套步骤思考。**方法是强参考；能力边界、执行状态和完成结果才由 Runtime（运行时）强制。**

<summary>展开：本文对几个工程名词的约定</summary>

| 本文称呼 | 指什么 |
| --- | --- |
| Harness System（运行支架系统） | 模型周围完整的运行环境 |
| Business Harness（业务运行支架） | 配装、排轴这类完整业务的责任边界 |
| Business Harness Version（业务运行支架版本） | 某项业务在一次发布中采用的方法、规则和完成条件 |
| Harness Runtime（运行支架运行时） | 加载这些声明、维护进度并执行硬检查的程序 |

这不是行业统一术语，只是为了避免把所有东西都叫 Harness。

</details>

## Harness（运行支架）和 Tools（工具）、Skills（技能包）、Workflow（工作流）的关系

这几组关系不用讲复杂。看它们怎样作用于 Agent 就够了：

| 概念 | 作用方式 | 它管什么 |
| --- | --- | --- |
| Tool（工具） | **强制能力** | 可以不调用；一旦调用，输入、执行和结果由程序定义 |
| Skill（技能包） | **非强制参考** | 提供一份可选的方法目录 |
| Workflow（工作流） | **强制路径** | 按确定顺序串联 Tools |
| Context（上下文） | **当前材料** | 把本轮判断需要的事实和任务状态摆到模型面前 |
| Prompt（提示词） | **模型说明** | 告诉模型目标和方法，不能替产品执行约束 |
| Business Harness（业务运行支架） | **方法层非强制、但强参考** | 让一套领域方法成为默认工作依据，并组织知识、Context、能力、状态与完成验证 |
| Runtime（运行时） | **强制边界与后果** | 检查能力、权限、状态、写入和真实结果 |

Harness 不能和 Tool、Skill、Workflow 算成同一层的部件，因为它会把这些东西和更多运行条件组织在一起。但从效果上比较，区别很清楚：

> **Skill 是“这里有一份资料”；Workflow 是“必须按这条路走”；Harness 则让一套领域方法成为强参考，但不把偏离方法本身判成错误。**

## 本项目 Harness（运行支架）的特性：先从“解一道题”说起

很多读者没开发过 Agent，但都做过题。

一道普通证明题给出条件和结论。解题的人要看懂题目，想起可用的定理，选择证明办法；走不通，就换一条路。

知识工程里有一个概念叫 Problem-Solving Method（问题求解方法，简称 PSM），讨论的就是“一类问题可以怎样解”。[知识工程对 PSM 的介绍](https://www.cs.vu.nl/~guus/papers/Schreiber07a.pdf)

Agent 的运行看起来也很像：理解用户的话，取得需要的信息，调用 Tools，处理返回结果；如果事情还没完成，就继续下一轮。

PSM 让这件事变得好懂，但它到这里就该停了。

> **PSM 不是 Harness，也没有智能；它只是借“怎样解一道题”这个熟悉视角，让读者看见 Harness 需要为智能体组织和管理题目、方法与临场判断。**

Harness 不替模型解题。它负责把这一类题需要的知识、方法、当下材料和可用能力准备好，也接住模型临场做出的动作，保存进度，守住边界，最后检查事情是不是真的完成了。

### 为什么不能强制？

同一句“帮我配一套装备”，合法的走法并不只有一条。

- 用户只想听建议，Agent 不必进入写入。
- 目标角色不清楚，Agent 可以先追问。
- 现有证据不够，Agent 可以先读攻略，也可以先看装备数据。
- 等用户确认时页面已经变化，原方案就要重新判断。

这些情况当然可以全写成分支。但分支越写越多，最后得到的是一条巨大的 Workflow，不是一个仍能临场判断的 Agent。

> **当你把这一切都写死，还有什么意义？到底是你在管这些问题，还是 Harness 在管？**

所以，Runtime 不该检查模型有没有逐步照抄推荐方法。模型换了一条路，只要没有越界，并且真实结果通过验证，这项任务仍然可以成功。

这就是这篇手记最想说的一句：

> **方法不强制，边界和后果强制。**

## Runtime（运行时）在整个 Agent（智能体）中扮演什么角色？

Harness 组织一类业务的完整工作条件，Runtime 负责让这些条件在每一次真实任务中生效。

它不会替模型判断哪套装备更好。它做的是另外一类事情：

| 时刻 | Runtime 做什么 |
| --- | --- |
| 模型判断前 | 识别业务、固定 Harness 版本、组装当前事实和任务状态、提供 Tools 与方法参考 |
| 模型提出动作后 | 检查 Schema（结构定义）、权限、确认、状态版本和写入范围，再决定执行或拒绝 |
| 动作执行后 | 刷新真实产品状态，保存进度，检查结果是否达到完成条件 |

一项配装可能跨过多轮对话，甚至停半天等用户确认。因此只保存 Session（会话）还不够。

本文把这一次可暂停、可恢复的具体工作叫作**业务事务**：Session 记得“聊过什么”，业务事务记得“事情做到哪里”。

比如用户确认前手动换了一件装备。聊天记录没有丢，但旧 Proposal（待确认方案）已经过期。Runtime 要拒绝旧写入，重新取得真实状态，再把变化交还给模型判断。

于是三者的关系很简单：

> **Harness 组织工作条件，Runtime 加载条件并强制其中的硬边界，Trace（执行轨迹）记录实际发生了什么。**

<details>
<summary>展开：Runtime 在配装事务中保存和检查什么</summary>

- 固定这次任务使用的 Business Harness 版本；
- 保存目标角色、队伍、当前配装和产品状态版本；
- 保存模型形成的候选、用户修正和待确认方案；
- 在真实修改前检查权限、确认对象、状态版本和写入范围；
- 修改后重新读取真实页面；
- 结果一致才完成；不一致就继续处理、恢复或失败。

OpenAI Agents SDK（OpenAI 智能体开发工具包）已经提供基础 Agent Loop（智能体循环）、会话记忆和 Human-in-the-loop（人在回路）；这里的 Runtime 还要补上业务路由、业务事务、Harness 版本和领域完成条件。[OpenAI Agents SDK：Running agents（运行智能体）](https://openai.github.io/openai-agents-python/running_agents/)

</details>

## 典型的垂直领域 Harness（运行支架）

垂直领域不是把 Agent 的名字换成行业名词。

它真正特殊的地方，是**同一类问题会反复出现**。

### 1、垂直领域的特性：解决实际问题

今天有人要配装，明天还会有人要配装。

每次的角色、队伍、页面状态和用户意图都不同。模型要读哪些材料、先问什么、怎样比较，也可能不同。

但这项业务承担什么责任，哪些来源可信，可以使用哪些能力，哪里不能写，怎样才算完成，却会反复出现。

**稳定的不是一条解题路线，而是一类问题的工作条件。**

<details>
<summary>展开：通用 Agent 与垂直领域 Agent 的工程重心</summary>

| 通用 Agent（智能体） | 垂直领域 Agent（智能体） |
| --- | --- |
| 问题范围开放，结构多由用户临时给出 | 问题集合相对稳定，可以提前建模 |
| 按任务临时寻找事实来源 | 可以预先规定每类事实的权威来源 |
| 每次临时规划 | 同类问题可以积累方法参考 |
| 完成条件随任务变化 | 可以沉淀写入范围和后置条件 |
| 主要迭代通用能力 | 还要迭代知识、方法和业务版本 |

这只是工程重心，不是绝对分类。Multi-agent（多智能体）解决的是几个 Agent 怎样分工，也不会自动补齐这些业务建模。

</details>

### 2、以本项目为例

再走一次开头那句话：

> 帮我给当前角色搭一套适合当前队伍的装备。

用户要的是完整配装，不是查询某件装备，也不是修改一个栏位。于是这项请求进入配装 Business Harness。

模型可能先确认“当前角色”是谁，也可能已经从页面取得答案；可能先看现有装备，也可能因为目标含糊而先追问。Harness 不固定这个顺序。

但它会把需要的条件组织起来：真实角色与队伍、当前配装、带来源的攻略、装备数据、公式、可用 Tools、写入边界，以及这次任务已经做到哪里。

模型据此形成方案。用户只要建议，可以到这里结束；用户要应用，Runtime 就在写入前检查确认和页面版本，写入后再读真实页面。Tool 返回 `success`，仍然不等于整项业务已经完成。

配装 Business Harness 对用户承担的完整承诺，可以写成一句话：

> **理解目标，基于可追溯的事实形成可解释方案；需要落到产品时，经用户确认，在配装边界内修改，并用真实产品状态证明结果。**

## 垂直领域的知识（游戏知识）

### 1、游戏知识和 Harness（运行支架）的关系

**Harness 使用知识，但不复制知识。**

它要说明的是：这类问题需要哪些知识，去哪里读，各自能证明什么。攻略更新不该迫使整个流程重写；方法调整也不该复制一份新的装备数据库。

<details>
<summary>展开：配装 Harness 使用的几类知识</summary>

| 知识角色 | 权威来源 | 用来做什么 |
| --- | --- | --- |
| 当前事实 | 真实产品状态 | 确定这次问题的起点 |
| 游戏事实 | Catalog（数据目录）、Formula Engine（公式引擎） | 核对装备与计算结果 |
| 策略知识 | 带来源和适用条件的攻略 | 形成候选、解释选择 |
| 本次决定 | 业务事务 | 保存修正、Proposal 和确认 |

**文字可以解释证据，不能代替证据。**

</details>

<details>
<summary>展开：为什么 Context 不该一次塞满</summary>

产品状态、Catalog、攻略、公式结果、业务事务和会话历史有不同的来源与寿命。业务 Harness 只声明需要什么，Runtime 再按本轮任务选择、压缩和刷新。

代码侧还可能保留身份、权限和任务引用；真正进入模型的，只是当前判断所需的信息。OpenAI Agents SDK 也区分 Local Run Context（本地运行上下文）与 Model-visible Context（模型可见上下文）。[OpenAI Agents SDK：Context management（上下文管理）](https://openai.github.io/openai-agents-python/context/)

</details>

### 2、问题：写死 Prompts（提示词）？实际遇到的迭代问题

项目早期把业务规则写进 Prompt，没有错。业务还没稳定时，这是验证想法最快的地方。

后来遇到一个很实际的问题：Skill 偶尔没有加载。

为了保险，我们把关键规则复制进固定 Prompt。接着又复制到 Tool Description（工具描述）和 Runtime。Prompt 越来越长，同一句“应用配装前必须确认”也到处都是。

这时问题已经不是“Prompt 太长”，而是：

> **一项业务行为被写在很多地方，却没有一个完整的责任主体。**

解法不是再写一份更大的 Prompt。方法和完成条件回到 Business Harness，状态与硬边界交给 Runtime 和 Product Kernel（产品内核）。Prompt 仍然可以迭代，只是不再充当所有业务规则的仓库。

<details>
<summary>展开：一项行为应该由谁负责</summary>

| 内容 | 主要负责人 |
| --- | --- |
| 当前页面和方案事实 | Host Kernel（宿主内核） |
| 业务目标、方法参考、知识角色和完成条件 | Business Harness（业务运行支架） |
| 带来源和适用条件的攻略 | 领域知识 |
| 单个能力的输入、输出、错误和执行 | Typed Tool（类型化工具） |
| 路由、事务、版本、Context 和阶段推进 | Harness Runtime（运行支架运行时） |
| 权限、审批、写入和真实后置检查 | Product Kernel（产品内核） |
| 实际发生了什么 | Trace（执行轨迹） |

[Harness Handbook](https://ruhan-wang.github.io/Harness-Handbook/#one-behavior-many-implementation-sites)（《运行支架手册》）把这类问题称为“一个行为，多个实现位置”。文件树告诉我们代码放在哪里；业务 Harness 和 Trace 让我们追问这项行为由谁负责、实际有没有发生。

</details>

### 3、Typed Tools（类型化工具）完成之后，迭代的是什么？

Typed Tools 完成，只表示 Agent 的手相对稳定了。

之后长期变化的，是它**怎样使用这些手解决业务**：配装方法会变，攻略会变，产品动作会增加，审批和状态检查也会调整。

把变化放回正确位置，项目才回答得了两件事：**这次行为为什么变了？只改配装，会不会误伤选人、排轴或 BUFF？**

<details>
<summary>展开：常见变化应该落在哪里</summary>

| 变化 | 主要修改位置 |
| --- | --- |
| 新增读取或写入能力 | Typed Tool 与产品接口 |
| Tool 参数、返回值或错误变化 | Tool Schema（工具结构定义） / Handler（处理器） |
| 装备数据或公式变化 | Catalog / Formula Engine（公式引擎） |
| 攻略结论和适用条件变化 | 领域知识 |
| 配装怎样分析、追问和形成候选 | 配装 Harness 版本 |
| 事务、阶段和版本怎样运行 | Harness Runtime |
| 权限、审批或写入范围变化 | Runtime / Product Kernel |
| 真实结果怎样验证 | 产品状态与 Trace |

</details>

## 拆解业务 → 建模：从方法走向可迭代体系

方法、知识和执行责任各自归位以后，还差一个边界。

如果全部塞进一个通用 Harness，改配装方法时，选人、排轴和计算都可能一起受影响。反过来，如果按字段或 Tool 切得太碎，又没有任何东西对“搭好一套装备”负责。

所以真正的问题不是“文件放在哪个目录”，而是：

> **哪一块能接住一件完整工作，而且可以独立修改？**

### 1、形象的说法：原子化 Harness（运行支架）

这里的“原子”不是最小文件、最小 Tool 或最小步骤，而是：

> **一类拥有完整业务责任、明确写入边界，并且可以独立版本化和完成验证的领域问题。**

用户会把它当成一件完整工作提出。它有自己的方法参考、知识角色、任务状态、能力范围和完成条件，也值得独立发布和回退。

按这个标准，本项目可以拆成五个 Business Harness：

| Business Harness（业务运行支架） | 它负责什么 | 完成证据 |
| --- | --- | --- |
| 选人 | 增加、删除、替换和调整队伍 | 真实页面中的当前队伍 |
| 配装 | 分析、比较、预览和应用角色配置 | 真实角色配置页 |
| 排轴 | 添加、移动、替换和应用技能按钮 | 真实时间轴 |
| BUFF | 查询、添加、替换和批量处理 BUFF | 真实按钮 BUFF 状态 |
| 计算统计 | 计算、比较、归因和解释 | 绑定方案与公式版本的结果 |

所以“当前角色”是实体，“适合当前队伍”是目标条件，武器和护甲是候选的一部分，`apply_loadout` 是 Tool；**“搭一套装备”才是一项完整业务。**

原子化保护的是完整责任和独立迭代，不是把系统切得越碎越好。

### 2、为什么需要版本？

版本不是为了给每段 Prompt 和每次 Tool Call（工具调用）编号。只有能独立接住一件事、也会独立变化的业务能力，才值得单独版本化。

配装 V1 已经形成待确认方案，用户离开了。第二天 V2 发布。用户回来点击“确认”时，这项任务不能偷偷换成 V2——他确认的是 V1 在当时事实下形成的方案。

所以任务开始时固定版本。新任务可以使用 V2，进行中的任务继续使用 V1，除非旧版本被明确撤销。

<summary>展开：业务定义、版本和事务分别记录什么</summary>

| 记录 | 负责什么 | 例子 |
| --- | --- | --- |
| 业务定义 | 稳定的责任、写入范围和完成条件 | 配装能改装备，不能改队伍成员 |
| Business Harness 版本 | 本次发布采用的方法参考、知识入口和异常处理 | 配装 V1、V2 |
| 业务事务 | 这一次的目标、证据、Proposal、确认和进度 | 正在给当前角色配装 |

### 3、业务拆分？No，是五项完整能力

如果只是把代码分成五个目录，事情没有完成。

每个 Business Harness 都要有自己的**业务责任、方法参考、知识角色、事务状态、能力范围、写入边界、完成验证和版本生命周期**。它们可以共享 Tools、知识和产品状态，但不能共享模糊的业务所有权。

跨业务请求也不需要一个“万能 Harness”。Manager（管理器）可以保存总体计划，再按当前目标创建需要的业务事务；每一项实际工作仍由自己的 Business Harness 负责。

最终得到的不是五份更长的 Prompt，而是**五项可以独立运行、换版、验证和追溯的领域问题求解能力**。

<details>
<summary>展开：字段位置和跨业务计划都不会取消业务边界</summary>

`selectedBuff` 可能保存在时间轴按钮上。排轴 Harness 负责按钮身份、位置和顺序；BUFF Harness 负责同一按钮上的 BUFF。字段住在同一个对象里，不代表两个业务可以互相修改对方的责任范围。

用户同时提出换人、配装、排轴、上 BUFF 和计算时，Manager 保存的是跨业务计划。下面只是一种可能的依赖顺序，不是 Harness 的默认定义：

```text
选人 → 配装 → 排轴 → BUFF → 计算
```

前一步失败、被拒绝或过期，后一步就不能继续消费旧结果。

## 先停在这里

绕了一圈，Harness 仍然是最合适的主名称。

因为它说的是方法、材料、能力、状态和约束围绕 Agent 形成的整体，而不是其中某一个部件。

它不替模型预先解完问题，也不要求模型走唯一的路。它给模型足够强的参考，再由 Runtime 守住能力、状态、执行和完成边界。

所以本篇最后留下的，不是一张配装流程图，而是一个判断：

> **垂直领域 Harness 把一类反复出现的业务问题，变成模型可以自主判断、系统可以持续管理、结果可以真实验证的完整能力。**

至于运行轨迹怎样反过来改进 Harness，又怎样形成自训练，留到下一篇单独说。
