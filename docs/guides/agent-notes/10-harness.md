# Harness：围绕 Agent 的求解环境

前面的篇章已经让 Agent 拿到了 Tools，也让这些能力可以通过 MCP 被外部 AI 调用。但能调用一组能力，不等于 Agent 已经知道怎样解决一类真实业务问题。

## 什么是 Harness？

Harness 是围绕模型搭建的一整套求解环境。它组织模型能看到什么、可以使用什么、怎样接触真实状态，以及怎样判断结果。

Harness 不替模型思考，也不要求模型只能走一条固定路径。

## 广义 Harness 会指什么？

广义的 Agent Harness 会包括 Prompts、Context、Skills、Tools、Memory、Agent Loop、权限、状态和结果验证。不同项目的边界并不完全相同，使用这个词时需要先给出自己的定义。

## Harness 和 Tools、Workflow 的关系

| 概念 | 是什么 | 和 Harness 的关系 |
| --- | --- | --- |
| Tool | 一个可执行动作 | Harness 决定当前提供哪些 Tools |
| Workflow | 一条明确的步骤路径 | Harness 可以提供流程参考，但不限定唯一路径 |
| Harness | 一类问题的完整求解环境 | 组织 Tools、流程、知识、状态和验证 |

Tool 解决“能做什么”，Workflow 解决“按什么步骤做”，Harness 解决“在什么业务环境里把问题做完”。

## Harness 和 Skills 的关系

| 对比项 | Skill | Harness |
| --- | --- | --- |
| 内容 | 知识、经验或操作方法 | 一类问题的完整求解环境 |
| 使用方式 | 模型自行理解和选择 | Runtime 按当前业务加载 |
| 是否强制 | 非强制 | 方法是强参考，执行边界可以强制 |
| 状态 | 通常不保存业务状态 | 可以绑定事务和真实产品状态 |

Skill 可以教模型怎样做，但不能保证模型一定采用。涉及权限、写入和结果正确性时，还需要 Harness、Runtime 和产品代码。

## Harness 和 Context 的关系

| 对比项 | Context | Harness |
| --- | --- | --- |
| 是什么 | 当前一轮模型看到的材料 | 准备这些材料的环境 |
| 内容 | 对话、状态、知识、Tool Result | 选择规则、Tools、事务和完成条件 |
| 生命周期 | 每轮都可能变化 | 可以跨很多轮和很多事务复用 |

Context 是当前材料，Harness 决定什么时候把哪些材料交给模型。

## Harness 和 Prompts 的关系

Prompt 是 Harness 向模型传递信息的一种方式，不是 Harness 本身。权限、Tool Gate、事务状态和结果验证还需要运行代码。

## 本项目的 Harness 特性

| 层次 | 内容 | 是否强制 |
| --- | --- | --- |
| 执行边界 | Tool、权限、审批、写入范围、版本和结果验证 | 强制 |
| 求解范式 | 怎样理解目标、读取证据、形成候选和处理修正 | 强参考，非强制 |

Runtime 可以拒绝越权 Tool 调用，却不规定模型必须怎样思考。

### 为什么方法不能强制？

Tool 调用和执行结果可以检查，模型得出结论的过程不能被同样检查。把唯一方法写死，也会让 Harness 退化成无法应对歧义和变化的固定 Workflow。

## 什么是 Runtime？

Runtime 是让 Agent 真正运行起来的程序。它连接用户、模型、Harness、Tools 和产品状态。

```text
用户请求
  → Runtime 加载 Harness、Context 和 Tools
  → 模型选择下一步
  → Tool 执行
  → Runtime 保存状态并继续
```

Harness 说明需要什么求解环境，Runtime 负责在一次真实事务中把它建立起来。Runtime 落实硬边界，同时把求解方法作为参考交给模型。

## 典型的垂直领域 Harness

### 垂直领域首先要解决实际问题

垂直领域 Agent 必须明确四件事：

- 当前操作的是哪个业务对象。
- 哪些数据才是事实。
- 哪些动作会改变产品。
- 什么结果才算完成。

通用 Agent 面向更宽的任务，背景和判断标准通常由用户临时提供；垂直领域 Agent 面对相对稳定的问题、对象、事实来源和完成条件。

Multi-agent 解决“由几个 Agent 分工”。垂直领域 Harness 解决“这项工作依赖什么”，两者不是同一个问题。

### 以本项目的配装业务为例

“给别礼配置一套 3+1 潮涌装备”大致需要：

1. 绑定目标角色和当前配装。
2. 读取装备事实、攻略和适用条件。
3. 生成一套完整候选。
4. 让用户预览、修正或确认。
5. 应用以后核对真实页面。

这是配装问题的求解范式，不是固定 Workflow。模型可以追问、比较方案，也可以在用户只想查询时停止在应用之前。

### 从知识工程看 Problem-Solving Method

知识工程中有一个概念：**Problem-Solving Method（PSM，问题求解方法）**。它把一类任务的求解方法，与具体领域知识和程序实现分开。[知识工程对 PSM 的介绍](https://www.cs.vu.nl/~guus/papers/Schreiber07a.pdf)

| 知识工程 | Agent 开发 |
| --- | --- |
| 任务：要解决什么问题 | 用户当前要完成什么业务 |
| 方法：通常怎样求解 | Harness 中的方法参考 |
| 领域知识：求解需要什么知识 | 按当前问题读取的游戏知识 |
| 程序实现：怎样运行 | Runtime、Tools 和产品代码 |

PSM 只解释 Harness 的方法层。Harness 还包括 Context、Tools、状态、权限和验证，所以二者不能画等号。

## 垂直领域的知识

### 游戏知识和 Harness 的关系

| 内容 | 来源 |
| --- | --- |
| 当前队伍和配装 | 产品真实状态 |
| 装备、技能和 BUFF 数据 | Catalog |
| 角色定位和配装建议 | 带条件的游戏知识 |
| 伤害结果 | 公式和计算引擎 |

知识提供证据，Harness 决定当前问题需要读取哪些证据。攻略不能覆盖产品事实，Harness 也不能自己编造游戏数据。

### 为什么不能把知识写死在 Prompts 里？

写进固定 Prompt 很直接，但迭代以后会出现：

- 同一条规则散落在 Prompt、Skill 和 Tool Description。
- 新旧内容同时存在，不知道以谁为准。
- 无关知识长期占用 Context。
- 知识失去来源、条件和独立版本。

Prompt 应该是当前内容的投影，不应该继续充当全部业务知识的唯一存放位置。

### Typed Tools 完成以后，迭代的是什么？

| 变化 | 迭代位置 |
| --- | --- |
| 产品能力变化 | Typed Tool |
| 游戏事实或公式变化 | Catalog 或公式 |
| 攻略和适用条件变化 | 领域知识 |
| 一类问题的求解方法变化 | 对应 Harness |
| 加载、事务和版本机制变化 | Runtime |

因此，主要迭代的是 Harness 和领域知识。Runtime 再把它们组装成当前一轮的 Prompt、Context 和 Tools。

## 从业务拆解到建模

建立 Harness 时，不先看有几个 Tools，而先看用户要完成哪一类完整问题：

- 最终结果是什么。
- 需要哪些事实和知识。
- 可以使用哪些能力。
- 可以修改什么。
- 怎样验证完成。

“3+1”是配装条件，不是一个 Harness；四件装备也不是四个 Harness。它们共同属于一个完整的配装问题。

### 原子化 Harness

原子化不是把步骤拆到最小，而是让一类完整业务可以独立求解、独立验证和独立迭代。

本项目把选人、配装、排轴、BUFF 和计算统计建模为五个原子 Harness。批量 BUFF 仍属于 BUFF，`3+1` 仍属于配装，不再继续拆分。

### 迭代与管理

每个原子 Harness 可以独立更新、启用、回退或撤销。新事务使用新版本，已经开始的事务继续使用原版本，避免规则在确认过程中改变。

### 这只是业务拆分吗？

不是。拆分以后还要为每类问题建立方法、知识、Tools、事务和验证，并让它们能够独立迭代。

最终得到的是一套**领域问题求解 Harness 及其可迭代体系**。
