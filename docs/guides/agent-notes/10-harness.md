# Harness：围绕 Agent 的求解环境

前面的篇章已经让 Agent 拿到了 Tools，也让这些能力可以通过 MCP 被外部 AI 调用。但能调用一组能力，不等于 Agent 已经知道怎样解决一类真实业务问题。Harness 讨论的正是 Tool 之外的这一层。

## 什么是 Harness？

Harness 是围绕模型搭建的一整套求解环境。它组织模型能看到什么、可以使用什么、怎样接触真实状态，以及执行以后怎样判断结果。

Harness 不替模型思考，也不要求模型只能走一条固定路径。它让模型的自主判断发生在一个与实际业务相连的环境中。

## 广义 Harness 会指什么？

Harness 原本就有“把主体连接到周围装置中”的含义。在 Agent 语境里，它经常被广义地用来指模型外围的整套工程结构，包括 Prompts、Context 组装、Skills、Tools、Memory、Agent Loop、权限、状态保存、结果验证和可观测记录。

不同项目对 Harness 的边界并不完全相同。有些项目只把模型调用和 Tool Loop 称为 Harness，有些项目还会把领域知识、业务状态和评测一起算进去。因此，讨论具体系统时需要先说明自己使用的是哪一种定义。

## Harness 和 Tools、Workflow 的关系

| 概念 | 它回答的问题 | 典型内容 | 是否规定求解路径 |
| --- | --- | --- | --- |
| Tool | Agent 能执行什么动作？ | 查询数据、创建草稿、保存配置、运行计算 | 不规定路径，只提供单项能力 |
| Workflow | 这项任务由哪些步骤和分支组成？ | 读取、生成、确认、应用，或一张显式流程图 | 通常会描述较明确的步骤关系 |
| Harness | Agent 在什么环境中解决这类问题？ | 业务边界、Context、知识、Tools、状态、权限和结果验证 | 可以提供强参考，但不必固定唯一路径 |

Tool 是动作，Workflow 是动作之间的一种组织方式，Harness 则是承载这些动作和路径的环境。同一个 Harness 可以提供同一组 Tools，却因为用户目标、当前状态和模型判断不同，走出不同的实际路径。

如果 Harness 把每一步和每个分支都预先固定，它当然可以包含一个确定性的 Workflow；但这只是 Harness 的一种实现，不是 Harness 的定义。

## Harness 和 Skills 的关系

| 对比项 | Skill | Harness |
| --- | --- | --- |
| 主要作用 | 向模型提供一组知识、经验或操作方法 | 为一类问题组织完整的求解环境 |
| 常见载体 | Markdown、说明文字、示例、脚本和参考资料 | 配置、状态、知识入口、Tool 投影、运行规则和验证机制 |
| 模型怎样使用 | 模型读取后自行理解、选择和执行 | Runtime 把其中与当前事务有关的部分交给模型 |
| 是否强制 | 非强制。即使被加载，文本本身也不能保证模型一定采用其中的方法 | 其中的方法可以是强参考；可执行边界可以通过 Runtime 和产品代码强制 |
| 能否独立拦截错误操作 | 不能 | 可以借助 Tool Gate、权限和产品状态检查拦截 |
| 是否保存业务状态 | 通常不负责 | 可以与事务、版本和真实产品状态绑定 |

Skill 的非强制性不是缺点。它适合把可复用经验交给模型，让模型根据当前问题决定是否采用。但当一条规则涉及真实写入、权限或结果正确性时，只把它写进 Skill 并不够。

Harness 可以读取 Skill，也可以把原本散落在 Skill 里的程序性业务知识纳入自己的方法参考。两者的区别不在于内容是不是 Markdown，而在于它们是否参与了完整业务环境的组织和运行。

## Harness 和 Context 的关系

| 对比项 | Context | Harness |
| --- | --- | --- |
| 是什么 | 当前一次模型请求真正能看到的内容 | 决定如何准备和使用这些内容的环境 |
| 典型内容 | 对话、当前方案、用户目标、知识片段、Tool Result | Context 选择规则、知识入口、事务状态、Tools 和完成条件 |
| 生命周期 | 会随每一轮请求变化 | 可以跨很多轮、很多事务复用 |
| 与当前业务的关系 | 是这一次求解使用的具体材料 | 决定哪些材料与当前业务有关 |

Context 是一轮请求里的实际材料，Harness 则决定材料怎样进入、何时更新、与哪一项真实业务绑定。把所有可能相关的内容一次性塞进 Context，不等于已经建立了 Harness。

## Harness 和 Prompts 的关系

Prompt 是 Harness 向模型表达信息的一种方式，但不是 Harness 本身。业务目标、方法参考和部分 Context 可以通过 Prompt 交给模型；Tool 可见性、权限、事务状态和真实结果验证则还需要运行代码来完成。

因此，Harness 的一部分最终会呈现为 Prompt，但不能因为模型最后看到了一段文字，就把整套 Harness 简化成一份巨型 Prompt。

## 本项目的 Harness 有什么特性？

DEF Workbench 中的 Harness 同时包含两种不同性质的内容：

| 层次 | 本项目中的内容 | 是否强制 |
| --- | --- | --- |
| 可执行边界 | 当前可以看到和调用哪些 Typed Tools、可以修改什么、是否需要审批、版本是否匹配、结果是否真实生效 | 强制 |
| 问题求解范式 | 怎样理解用户目标、需要读取哪些证据、怎样形成候选、什么时候追问或重新规划、怎样说明结果 | 强参考，非强制 |

Tools 的边界可以强制。Runtime 可以只暴露当前允许使用的 Tool，也可以在执行前再次检查调用是否越权；产品层还可以继续检查 Permission、Approval、Schema、CAS 和最终状态。

实际解决业务的方法不能用同一种方式强制。Runtime 能检查模型提交了什么动作以及动作产生了什么结果，却不能规定模型必须怎样得出这个动作。用户的表达可能有歧义，产品状态会变化，同一个目标也可能存在多种有效方案；如果把唯一推理路径写死，Harness 就会退化成一条脆弱的 Workflow。

## 什么是 Runtime？

Runtime 是真正让 Agent 运转起来的执行系统。它接收用户请求，准备一次模型调用，处理 Tool Call 和 Tool Result，保存 Session 与事务状态，然后决定是否继续下一轮。

在一个完整 Agent 中，模型负责理解和生成，Tools 负责接触外部世界，Harness 负责描述当前求解环境，Runtime 则把这些部分接到一起：

```text
用户请求 + 当前产品状态
          ↓
Runtime 选择并加载 Harness
          ↓
绑定 Context、知识和当前可用 Tools
          ↓
模型判断下一步
          ↓
Tool 执行、产品校验、结果返回
          ↓
Runtime 更新状态并准备下一轮
```

Harness 与 Runtime 不是同一个概念。Harness 说明这一类业务需要什么环境，Runtime 负责在某一次真实事务中把这个环境建立起来。Runtime 可以加载不同业务、不同版本的 Harness；同一份 Harness 也可以被 Runtime 用在许多次独立事务中。

本项目还需要区分软硬两侧。Runtime 会实际落实 Tool 和产品状态的硬边界，同时把问题求解范式作为强参考提供给模型；它不会把这份方法参考伪装成对模型内部思考过程的强制控制。

## 典型的垂直领域 Harness

### 垂直领域首先要解决实际问题

垂直领域 Agent 的目标不是展示“模型会调用工具”，而是完成某一类可以被业务理解和验证的工作。它需要知道当前操作的是哪个对象、哪些数据才是事实、什么动作会改变产品，以及什么结果才算完成。

通用 Agent 也能解决实际问题，但它通常面向范围很宽的任务，依赖用户临时提供背景和判断标准。垂直领域 Agent 面对的是相对稳定的问题类型、领域对象、事实来源和完成条件。它需要把这些稳定结构沉淀在模型之外，而不是每次都让用户从头解释。

Multi-agent 讨论的是由几个 Agent 分工、怎样交接任务。垂直领域 Harness 讨论的是无论由一个还是多个 Agent 执行，这项领域工作依赖哪些事实、能力和完成条件。增加 Agent 数量不会自动补上缺失的领域建模。

### 以本项目的配装业务为例

DEF Workbench 需要处理选人、配装、排轴、BUFF 和计算统计等业务。以“给别礼配置一套 3+1 潮涌装备”为例，Agent 不能只看见几个修改字段的 Tools，它还需要完成一整段业务理解：

1. 确认目标角色、当前队伍、当前配装和用户限制。
2. 理解“3+1”和“潮涌套”在当前业务里的含义。
3. 从角色攻略、装备目录和适用条件中读取相关证据。
4. 形成一套完整候选，而不是把四件装备拆成互不相关的四次修改。
5. 让用户查看预览，并根据用户的质疑或修正重新规划。
6. 用户明确确认后，通过 Typed Tool 应用原来的候选。
7. 回到真实 Operator Config 页面核对最终结果。

这组步骤表达的是配装问题的一般求解范式，不是一条必须逐字执行的固定 Workflow。如果用户只是询问“3+1 是什么”，流程不需要进入修改；如果目标角色不明确，模型可以先追问；如果用户同时要求比较两套装备，模型也可以先组织比较。Harness 提供的是业务方向和边界，模型仍然负责决定这次问题的具体路径。

### 从知识工程看 Problem-Solving Method

知识工程中有一个已有概念：**Problem-Solving Method（PSM，问题求解方法）**。它描述一类知识密集型任务可以怎样被求解，包括需要哪些知识角色、经过哪些推理活动，以及方法成立时依赖哪些假设。它的重要思想之一，是把可复用的求解方法与具体领域知识、具体程序实现分开。[知识工程对 PSM 的介绍](https://www.cs.vu.nl/~guus/papers/Schreiber07a.pdf)和 [UPML 对 Task、PSM 与 Domain Model 的划分](https://oro.open.ac.uk/23736/)都体现了这种分层。

配装业务也可以用这个视角理解：

- **任务**：为某个角色形成并应用满足目标的配装。
- **领域知识**：角色定位、装备属性、套装条件和攻略判断。
- **求解方法**：绑定当前状态，读取相关证据，生成完整候选，处理修正，应用并验证。

| 知识工程关注的 PSM | 现代 Agent 开发中的 Harness |
| --- | --- |
| 描述一类任务的可复用求解方法 | 把模型放进可以解决这类任务的完整环境 |
| 强调任务、方法与领域知识的分离 | 还要连接 Context、Tools、状态、权限、版本和真实结果 |
| 首先是一种知识层的建模概念 | 还包含实际运行和产品工程问题 |
| 说明“怎样求解” | 同时处理“能够做什么”和“做完怎样验证” |

因此，PSM 可以帮助我们理解 Harness 里的方法层，但它不是 Harness 的同义词。Harness 也不是“被强制执行的 PSM”：其中的问题求解方法仍然是强参考，真正被代码强制的是外部可观察的能力和产品边界。

## 垂直领域的知识

### 游戏知识和 Harness 的关系

游戏知识为 Agent 提供求解所需的证据，Harness 决定在什么业务、什么目标和什么条件下读取这些证据。两者需要合作，但不能混成一份没有边界的说明文字。

本项目至少存在几类不同来源：

| 内容 | 应该来自哪里 | 在求解中的作用 |
| --- | --- | --- |
| 当前队伍、当前配装、当前按钮状态 | Checkout 和产品真实状态 | 说明现在实际是什么 |
| 装备、技能和 BUFF 的基础数据 | 产品 Catalog | 提供可核对的基础事实 |
| 角色定位、配装建议和适用场景 | 带来源和适用条件的游戏知识 | 提供分析与推荐依据 |
| 伤害结果和统计口径 | 公式与计算引擎 | 提供可重复计算的结果 |

Harness 不应该自己编造这些事实，也不应该让攻略覆盖当前产品状态。它负责根据当前问题找到最小必要知识，并让模型知道证据的来源和适用条件。

### 为什么不能把知识都写死在 Prompts 里？

项目早期把流程和知识写进固定 Prompt 很直接：模型每次都能看到，也不需要额外的加载系统。但随着业务增加，同一条规则开始同时出现在固定 Prompt、Host Prompt、Skill、旧 Harness 和 Tool Description 中。

这会产生几个实际问题：

- 一条规则修改以后，需要在多个位置同步更新。
- 新旧版本可能同时进入 Context，模型只能猜哪一条有效。
- 与当前业务无关的知识也会占用 Context。
- 攻略结论的来源、适用角色和适用版本容易丢失。
- 修改某一个业务的方法时，可能意外影响其他业务。
- Tool Description 开始承担流程教学，不再只是描述 Tool 本身。

例如，“3+1 潮涌套”是一项配装条件，不应该被写进全局 Prompt 后永久影响所有业务；它也不应该因为包含四件装备，就复制成四段相同流程。Harness 应当在进入配装事务后读取对应知识，再让模型结合当前角色和真实 Catalog 判断。

### Typed Tools 完成以后，迭代的是什么？

Typed Tools 解决的是能力接口：参数是什么、返回什么、会产生什么副作用，以及失败时怎样表达。接口稳定以后，项目仍会持续变化，但不同变化应该回到不同位置：

| 发生的变化 | 应该迭代什么 |
| --- | --- |
| 产品新增一种读取或修改能力 | Typed Tool |
| 装备数值、技能事实或公式变化 | Catalog、公式或对应事实源 |
| 攻略结论和适用条件变化 | 领域知识 |
| 配装问题的识别、证据使用、候选组织或纠错方式变化 | 配装 Harness 的版本 |
| Tool 投影、事务保存和版本加载机制变化 | Runtime |
| 当前一轮模型最终看到的文字变化 | 通常是上述内容重新组装后的 Prompt |

所以，接下来迭代的不是“Prompt 还是 Harness”这个二选一问题。真正作为来源被维护的是各业务的 Harness 版本和独立领域知识；Runtime 再根据当前事务，把需要的部分组装成这一轮的 Context、Prompt 和 Tool 集合。

Prompt 仍然存在，但它更像一次运行时生成的视图，而不再是全部业务知识和流程规则的唯一存放位置。

## 从业务拆解到领域问题建模

### 拆的不是 Tool，也不是每一个步骤

建立 Harness 之前，需要先问的不是“项目里有几个 Tools”，而是“用户正在要求完成哪一类完整业务问题”。

以配装为例，读取 Catalog、生成候选、创建预览和应用配置可以使用多个 Tools；它们仍然共同服务于一个完整的配装问题。“3+1”只是候选需要满足的条件，四件装备只是同一份方案里的对象，都不应该各自成为 Harness。

反过来，选人和配装虽然可能出现在同一句用户请求中，却拥有不同的业务责任和写入结果。选人决定队伍成员，配装决定角色的装备与配置；前一项完成以后，后一项需要使用新的队伍状态重新开始自己的事务。

这种建模需要依次确认：

1. 用户最终想得到什么完整结果。
2. 解决问题必须绑定哪些当前事实。
3. 需要哪些领域知识和可执行能力。
4. 这类业务可以直接修改什么，不能修改什么。
5. 什么真实结果才表示已经完成。
6. 完成以后会让哪些下游结论过期或需要重算。

这正是前面 PSM 视角在 Agent 工程中的实际用途：它帮助我们按完整问题识别方法和知识角色，而不是按文件、实体或 Tool 名称切割系统。

### 一个更形象的说法：原子化 Harness

“原子化”不是追求最小文件或最少步骤，而是让一类完整业务责任拥有自己的边界。一个原子 Harness 应当能够独立说明自己解决什么问题、读取什么知识、使用什么能力、产生什么结果，以及怎样判断完成。

在本项目中，选人、配装、排轴、BUFF 和计算统计分别对应五类领域问题。批量添加 BUFF 仍然属于 BUFF 问题，`3+1` 仍然属于配装条件，生成统计报表仍然属于计算统计；这些操作形式和领域名词不会继续裂变出新的 Harness。

### 迭代与管理

原子化以后，每项业务可以独立更新自己的方法参考，而不必重新发布一个包含所有业务的巨型 Prompt 或整包 Harness。新的业务事务使用当前启用的版本，已经开始的事务继续使用自己开始时绑定的版本，避免用户确认方案时求解规则已经在中途改变。

一个版本还可以独立校验、启用、回退或撤销。这样管理的不是几份静态提示词，而是一套能够跟随业务持续演进的问题求解环境。

### 这只是业务拆分吗？

不是。普通业务拆分容易停在模块、接口和数据所有权上；这里还需要描述 Agent 怎样理解目标、使用哪些知识、调用哪些能力、保存什么事务状态，以及怎样确认真实结果。

最终得到的不是五段固定 Workflow，也不是五组 Tool 目录，而是五个**领域问题求解 Harness，以及管理它们持续迭代的体系**。

## 开发与训练（未完待续）

走到这里，许多看起来像“模型能力不足”的问题，其实可以继续拆开：是模型本身不会推理，还是它没有拿到正确事实？是缺少可执行能力，还是业务方法只写在一段互相冲突的 Prompt 里？是需要训练模型，还是应该继续开发 Harness？

训练改变模型本身，Harness 开发改变模型工作的环境。两者怎样分工，留到下一部分继续。
