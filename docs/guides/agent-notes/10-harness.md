# 篇章 11：Harness（运行支架）

前面已经给 Agent（智能体）接上了 Typed Tools（类型化工具）。

它能读取当前队伍和配装，能修改装备、编辑排轴、添加 BUFF（增益效果），也能调用真实公式完成计算。到这里，Agent 算是有手了。

然后用户说：

> 帮我给当前角色搭一套适合当前队伍的装备。

这句话没有要求调用哪个 Tool（工具），也没有给出固定步骤。

它只给了一个结果：一套真正适合当前队伍的配装。

接着才轮到这些问题：

- “当前角色”到底是谁，在队伍里承担什么作用？
- 这个角色现在穿什么，队伍里还缺什么？
- 哪些攻略适用于当前队伍？
- 哪些装备真的存在，数值是什么？
- 用户只是想听建议，还是准备修改？
- 等用户确认时，原来的方案有没有被别人改过？
- Tool 说成功以后，真实页面到底变没变？

这些问题不能从 Tool 列表里直接得到答案。它们共同决定的是：**这个 Agent 怎样完成一项真实业务。**

当然也可以写一个 `auto_loadout`，把整套配装逻辑塞进代码。这样很快，但 Agent 学会的只是调用配装程序，并没有学会处理配装问题。

就像给一个只会读稿的人接上装备查询和修改 Tools。他有手了，仍然不会做攻略。因为 Prompt（提示词）还在教他读稿，Skill（技能包）里是读稿经验，Context（上下文）里也只有稿件。

那把 Prompt 里的“读稿”改成“做攻略”呢？

也不够。旧方法可能还留在 Host Prompt（宿主提示词）、Skill 和 Tool Description（工具描述）里；权限、审批、任务进度和真实产品状态，也不会跟着一句 Prompt 一起变化。

问题于是从“改哪一句 Prompt”，变成了：

> 模型周围这整套工作条件，究竟由谁组织？

这就是这篇手记里的 Harness。

## 什么是 Harness（运行支架）？

先给一个不严谨、但好理解的说法：

> **Harness 是模型的工作台。**

模型负责判断下一步。Harness 把这一刻需要的材料和能力摆到桌上，接住模型提出的动作，再处理动作产生的结果。

  <div><small>判断下一步</small><strong>模型</strong></div>
  <b>→</b>
  <div class="accent"><small>准备 · 约束 · 反馈</small><strong>Harness</strong></div>
  <b>→</b>
  <div><small>产生真实结果</small><strong>Tools + 产品</strong></div>
</div>

再说准确一点：

> **Harness 是围绕模型建立的运行机制。**

它决定模型在什么信息和能力条件下工作，模型输出怎样被系统接收、执行、拒绝、记录和反馈，一件尚未完成的工作又怎样继续下去。

所以 Harness 不是单独一份 Prompt，也不是一组 Tools。

同一个模型接上不同 Harness，会表现得像不同的 Agent。不是模型突然变聪明了，而是它醒来以后，看见的事实、拿到的能力、受到的限制和面对的完成标准都变了。

## 广义 Harness（运行支架）会指什么？

Harness 没有一条被所有框架共同采用的边界。

最窄的说法，可能只指 System Prompt（系统提示词）和 Tool Loop（工具调用循环）；更宽的说法，会把 Context、Memory（记忆）、Skills、状态、权限、沙箱、审批和结果检查都算进去。

因此，看到别人讨论 Harness，第一件事不是争定义，而是先问：

> 他把边界画到了哪里？

这篇采用宽一点的说法。广义 Harness 是模型与真实环境之间的**运行支架**：它让模型能够持续取得信息、提出行动、接收反馈并延续状态。[AI Harness Engineering](https://arxiv.org/abs/2605.13357)（《人工智能运行支架工程》）也把 Agent 能力放在“模型—Harness—环境”这个整体中讨论。

但后面还会反复出现几个相近的词，这里先约定一下：

| 本文称呼 | 指什么 |
| --- | --- |
| Harness（运行支架）系统 | 模型周围完整的运行环境 |
| 业务 Harness（运行支架） | 配装、排轴这类完整业务的责任边界 |
| 业务 Harness（运行支架）版本 | 某项业务在一次发布中采用的方法、规则和完成条件 |
| Harness Runtime（运行支架运行时） | 加载这些声明、维护进度并执行硬检查的程序 |

这不是行业统一术语，只是为了避免把所有东西都叫 Harness。

## Harness（运行支架）和 Tools（工具）、Workflow（工作流）的关系

**Tool 负责一个动作，Workflow 负责一段必须固定的路径。** 业务 Harness 使用它们，但负责的是从理解问题到验证结果的完整工作。

## Harness（运行支架）和 Skills（技能包）的关系

**Skill 是可复用的方法材料。** Harness 可以加载它；但“加载”不等于“照做”，硬边界仍要由程序执行。

## Harness（运行支架）和 Context（上下文）的关系

**Context 是这一轮摆到模型面前的材料。** Harness 声明需要什么，Runtime 负责从产品事实、知识和任务状态中组装。

## Harness（运行支架）和 Prompts（提示词）的关系

**Prompt 负责告诉模型，不负责替产品上锁。** 目标和方法可以写给模型，权限、审批和结果检查仍留在程序里。

几组关系放在一起，就是这张表：

| 概念 | 最短解释 | 和业务 Harness 的关系 |
| --- | --- | --- |
| Tool（工具） | 一个动作 | Harness 使用它读取、计算或修改 |
| Workflow（工作流） | 一段固定路径 | Harness 把必须确定的控制关系交给它 |
| Skill（技能包） | 一份可复用的方法材料 | Harness 可以加载它，但“加载”不等于“服从” |
| Context（上下文） | 本轮摆到模型面前的材料 | Harness 声明需要什么，Runtime 负责组装 |
| Prompt（提示词） | 写给模型看的说明 | 它传达目标和方法，不能代替程序约束 |
| Runtime（运行时） | 让声明运行起来的程序 | 它维护任务、执行检查，并把结果送回模型 |

一句话：**Tool 是手，Workflow 是固定动作，Skill 是参考手册，Context 是桌面材料，Prompt 是说明；Harness 负责组织一整类工作。**

## 本项目 Harness（运行支架）的特性

上面的关系只是坐标。真正的问题仍然是：

> 一个 Agent 到底怎样解决一类配装问题？

这里可以提前引入一个老概念：**Problem-Solving Method（问题求解方法，简称 PSM）。**

不严谨地说，PSM 就是**一类题的解题方法**。它把任务、方法和知识分开讨论。[知识工程对 PSM 的介绍](https://www.cs.vu.nl/~guus/papers/Schreiber07a.pdf)

| PSM 概念 | 配装里的对应物 |
| --- | --- |
| Task（任务） | 给角色完成一套适合当前队伍的配装 |
| Problem Instance（问题实例） | 这次的角色、队伍、限制和当前产品状态 |
| Method（方法） | 理解目标、读取证据、形成候选、比较和修正 |
| Domain Knowledge（领域知识） | 攻略、装备数据、公式和战斗约定 |

这张表很重要。它说明了两件事：

1. **配装不是一串 Tool Call（工具调用）。** Tool 只是解题时用到的能力。
2. **攻略也不是配装方法。** 攻略和装备数据是方法使用的知识。

但 PSM 还不够。它解释“这类题可以怎样解”，不负责模型循环、任务状态、权限、审批和真实结果验证。

所以本项目里的关系是：

> **PSM 描述解题方法；业务 Harness 把方法接到知识、Context、Tools、状态和完成条件上。**

这也自然分出了两条线：

- **求解方法**：理解目标、查看现状、读取知识、形成候选、追问或重做。
- **硬边界**：确认有效、版本一致、写入合法、真实结果通过验证。

前一条给模型留下判断空间，后一条由程序执行。本项目的业务 Harness 还必须能够**跨回合保存任务**、**用真实产品状态证明完成**，并且**按业务独立发布和回退**。

### 为什么不能强制？

**不能强制的是唯一的解题路径，不是权限和写入边界。**

用户可能只要建议，也可能要直接修改；角色不清楚时要先问，证据不足时要先查，页面已经变化时又要重新规划。把这些情况硬压成一条 Workflow，只会得到一条不断长出例外分支的流程。

程序真正能强制的，是可观察、会产生后果的部分：当前阶段可以调用什么、参数是否合法、用户确认了哪份方案、方案是否过期、写入有没有越界、页面最终是否真的改变。

**方法可以灵活，越界不可以；模型说完成不算，真实结果才算。**

<details>
<summary>展开：配装中可以被强制检查的内容</summary>

- 当前阶段允许出现哪些 Tools；
- Tool 参数是否满足 Schema（结构定义）；
- 修改是否越过配装业务的写入范围；
- 用户确认的是否仍是当前 proposal（待确认方案）；
- proposal 绑定的产品版本是否仍然有效；
- 写入后真实页面是否满足后置条件。

</details>

## Runtime（运行时）在整个 Agent（智能体）中扮演什么角色？它和 Harness（运行支架）是什么关系？

一份 Harness 声明不会自己跑起来。一项配装又可能跨过多轮对话，甚至停半天等用户确认。

本文把这一次可暂停、可恢复的具体工作叫作**业务事务**。负责让它继续运转的程序，就是 **Harness Runtime**。

| 名词 | 记住什么 |
| --- | --- |
| 业务 Harness（运行支架）版本 | 这类问题现在采用哪套方法和规则 |
| 业务事务 | 这一次问题已经做到哪里 |
| Runtime（运行时） | 加载版本、维护事务、组装 Context、执行硬检查 |
| Trace（执行轨迹） | 实际看过什么、做过什么、凭什么完成 |

这里最容易混的是 Session（会话）和业务事务：**Session 记得“聊过什么”，业务事务记得“事情做到哪里”。**

比如用户确认前手动换了一件装备。聊天记录没有丢，但旧 proposal 已经过期。Runtime 要拦住写入、重新读取，再让模型规划。

所以三者的关系可以收成一句：

> **Harness 声明应该发生什么，Runtime 让它发生，Trace 记录实际发生了什么。**

<details>
<summary>展开：一次配装事务怎样运行</summary>

1. 识别为配装业务，创建事务并固定 Harness 版本；
2. 绑定当前角色、队伍、配装和产品版本；
3. 按当前阶段组装 Context，并提供相应 Tools；
4. 保存模型形成的候选和待确认 proposal；
5. 用户确认后，检查权限、确认对象、产品版本和写入范围；
6. 执行修改，再读取真实页面；
7. 结果一致才完成；不一致则失败、恢复或重新规划。

OpenAI Agents SDK（OpenAI 智能体开发工具包）已经提供基础 Agent Loop（智能体循环）、会话记忆和 Human-in-the-loop（人在回路）；这里的 Runtime 还要补上业务路由、业务事务、Harness 版本和领域完成条件。[OpenAI Agents SDK：Running agents（运行智能体）](https://openai.github.io/openai-agents-python/running_agents/)

</details>

## 典型的垂直领域 Harness（运行支架）

通用 Agent 当然也能解决真实问题。垂直领域真正特殊的地方，是**同一类问题会反复出现**。

今天有人要配装，明天还会有人要配装。角色、事实来源、常用方法、写入范围和完成条件不必每次从零发明，因此值得做成一项可以持续迭代的业务能力。

### 1、垂直领域的特性：解决实际问题

所谓垂直，不是把 Agent 的名字换成行业名词，而是问题集合相对稳定，所以能够提前沉淀**权威事实源、PSM、写入边界和完成证据**。

<details>
<summary>展开：通用 Agent 与垂直领域 Agent 的工程重心</summary>

| 通用 Agent（智能体） | 垂直领域 Agent（智能体） |
| --- | --- |
| 问题范围开放，结构多由用户临时给出 | 问题集合相对稳定，可以提前建模 |
| 按任务临时寻找事实来源 | 可以预先规定每类事实的权威来源 |
| 每次临时规划 | 同类问题可以积累 PSM |
| 完成条件随任务变化 | 可以沉淀写入范围和后置条件 |
| 主要迭代通用能力 | 还要迭代知识、方法、版本和领域评测 |

这只是工程重心，不是绝对分类。Multi-agent（多智能体）解决的是几个 Agent 怎样分工，也不会自动补齐这些业务建模。

</details>

### 2、以本项目为例

再走一次开头那句话：

> 帮我给当前角色搭一套适合当前队伍的装备。

**先识别业务。** 用户要的是完整配装，不是查询某件装备，也不是修改一个栏位。这项请求进入配装 Harness。

**再绑定现状。** 当前角色是谁、处在哪支队伍、现在穿什么、页面是什么版本。没有这些事实，“适合当前队伍”只是一句空话。

**然后使用知识。** 攻略说明角色定位和队伍需求，Catalog（数据目录）证明装备名称与数值，公式负责可复现的比较。它们各管一件事，不能互相代替。

**接着形成 proposal（待确认方案）。** 四件装备放在一起只是组合；绑定目标、限制、理由和产品版本以后，才是一份可以确认的方案。

**最后处理修改。** “配件换一个”是修正，“就用这套”才是确认。Runtime 在写入前检查方案是否仍然有效，写入后再读真实页面。Tool 返回 `success`，不等于整项业务已经完成。

具体过程可以长，也可以短。真正不变的是：

> **目标清楚，事实有来源，候选完整，修改经确认，写入受限制，结果可验证。**

### 3、从 PSM（问题求解方法）到业务 Harness（运行支架）

现在 PSM 和 Harness 的衔接就清楚了。

PSM 给出一类配装问题的**解题骨架**；业务 Harness 再补上这套方法依赖的**知识入口、可用 Tools、任务状态、硬边界、完成证据和版本**。

它不是用 Harness 取代 PSM，而是让 PSM 能在真实产品里工作：

```text
PSM
+ 领域知识的使用约定
+ Context 与 Tools
+ 业务事务与 Runtime 约束
+ 完成验证
= 可以运行和迭代的业务 Harness
```

方法已经有了。下一问自然是：**它使用的知识放在哪里？**

## 垂直领域的知识（游戏知识）

有了配装方法，不代表 Agent 已经知道当前角色需要什么。它仍要读取当前配装、装备数值、攻略和计算结果。

### 1、游戏知识和 Harness（运行支架）的关系

**Harness 使用知识，但不复制知识。**

它需要声明的是：这类问题需要哪些知识、去哪里读取、各自扮演什么角色，以及事实变化后旧方案是否失效。

| 知识角色 | 权威来源 | 用来做什么 |
| --- | --- | --- |
| 当前事实 | 真实产品状态 | 确定这次问题的起点 |
| 游戏事实 | Catalog、公式引擎 | 核对装备与计算结果 |
| 策略知识 | 带来源和条件的攻略 | 生成候选、解释选择 |
| 本次决定 | 业务事务 | 保存修正、proposal 和确认 |

攻略更新，不该迫使整个流程重写；配装方法调整，也不该复制一份新的装备数据库。**文字可以解释证据，不能代替证据。**

<details>
<summary>展开：为什么 Context 不该一次塞满</summary>

产品状态、Catalog、攻略、公式结果、业务事务和会话历史有不同的来源与寿命。业务 Harness 只声明需要什么，Runtime 再按本轮任务选择、压缩和刷新。

代码侧还可能保留身份、权限和任务引用；真正进入模型的，只是当前判断所需的信息。OpenAI Agents SDK 也区分 Local Run Context（本地运行上下文）与 Model-visible Context（模型可见上下文）。[OpenAI Agents SDK：Context management（上下文管理）](https://openai.github.io/openai-agents-python/context/)

</details>

### 2、问题：写死 Prompts（提示词）？实际遇到的迭代问题

项目早期把业务规则写进 Prompt，没有错。流程没稳定时，Prompt 是验证想法最快的地方。

后来为了避免 Skill 漏加载，越来越多的业务教学进入固定 Prompt：识别业务、读取知识、安排 Tools、等待确认、解释失败、判断完成。它慢慢变成了一套藏在自然语言里的业务程序。

**问题不只是 Prompt 长，而是这套业务程序没有成为 Runtime 能识别、保存、约束和独立更新的对象。**

于是同一句“应用配装前必须确认”，同时出现在 Prompt、Skill、Tool Description 和 Runtime 里。每一处都像负责人，又没有一处说得清最终由谁负责。

解法不是再写一份更大的 Prompt，而是把方法提炼为业务 Harness，把状态和硬边界交给 Runtime 与 Product Kernel（产品内核）。Prompt 仍然迭代，只是不再充当所有业务规则的仓库。

<details>
<summary>展开：一项行为应该由谁负责</summary>

| 内容 | 主要负责人 |
| --- | --- |
| 当前页面和方案事实 | Host Kernel（宿主内核） |
| 业务目标、PSM、知识角色和完成条件 | 业务 Harness（运行支架） |
| 带来源和适用条件的攻略 | 领域知识 |
| 单个能力的输入、输出、错误和执行 | Typed Tool（类型化工具） |
| 路由、事务、版本、Context 和阶段推进 | Harness Runtime（运行支架运行时） |
| 权限、审批、写入和真实后置检查 | Product Kernel（产品内核） |
| 实际发生了什么、是否符合预期 | Trace / Judge（评估器） |

[Harness Handbook](https://ruhan-wang.github.io/Harness-Handbook/#one-behavior-many-implementation-sites)（《运行支架手册》）把这类问题称为“一个行为，多个实现位置”。文件树告诉我们代码放在哪里；业务 Harness 和 Trace 让我们追问这项行为由谁负责、实际有没有发生。

</details>

### 3、Typed Tools（类型化工具）完成之后，迭代的是什么？

Typed Tools 完成，只表示 Agent 的手相对稳定了。之后长期变化的，是它**怎样使用这些手解决业务**。

调整配装方法，改配装 Harness；攻略变化，改领域知识；产品新增动作，改 Tool；审批和版本检查变化，改 Runtime 或 Product Kernel。

把变化放回正确位置，项目才回答得了两件事：**这次行为为什么变了？只改配装，会不会误伤选人、排轴或 BUFF？**

<details>
<summary>展开：常见变化应该落在哪里</summary>

| 变化 | 主要修改位置 |
| --- | --- |
| 新增读取或写入能力 | Typed Tool 与产品接口 |
| Tool 参数、返回值或错误变化 | Tool Schema（工具结构定义） / Handler（处理器） |
| 装备数据或公式变化 | Catalog / Formula Engine（公式引擎） |
| 攻略结论和适用条件变化 | 领域知识 |
| 配装怎样分析、追问和形成候选 | 配装 Harness 版本 |
| 事务、阶段和版本怎样运行 | Harness Runtime |
| 权限、审批或写入范围变化 | Runtime / Product Kernel |
| 怎样判断系统做对没有 | Trace、评测与真实 UI（用户界面）验证 |

</details>

## 拆解业务 → 建模：从方法走向可迭代体系

方法、知识和执行责任各自归位以后，下一问不是“文件放在哪个目录”，而是：

> **什么东西能完整负责一项业务，也能独立修改？**

### 1、迭代与管理

配装 V1 已经形成待确认方案，用户离开了。第二天 V2 发布。用户回来点击“确认”时，这项任务能不能偷偷换成 V2？

**不能。**

用户确认的是 V1 在当时事实下形成的方案。中途换方法，相当于让他确认另一件事。

因此至少要分清三种记录：

| 记录 | 负责什么 | 例子 |
| --- | --- | --- |
| 业务定义 | 稳定的边界、写入范围和完成条件 | 配装能改装备，不能改队伍成员 |
| Harness 版本 | 本次发布采用的方法、知识入口和异常处理 | 配装 V1、V2 |
| 业务事务 | 这一次的目标、证据、proposal、确认和阶段 | 正在给当前角色配装 |

任务开始时固定 Harness 版本。新任务使用新版本，进行中的任务继续用旧版本，除非旧版本被明确撤销。这样配装可以单独热更新，也不会把整个 Session 永远锁在同一套大 Prompt 上。

<details>
<summary>展开：管理系统还需要做什么</summary>

- 注册、启用、回退或撤销业务版本；
- 把新请求路由到正确业务；
- 为一次事务固定 Harness 版本和产品状态；
- 保存可暂停、可恢复的任务进度；
- 让 Trace 回到当时真正运行的版本。

</details>

### 2、形象的说法：原子化 Harness（运行支架）

这里的“原子”不是最小文件、最小 Tool 或最小步骤，而是：

> **一类拥有完整业务责任，并且可以独立版本化和完成验证的领域问题。**

用户会把它当成一件完整工作提出；它有输入、输出、PSM、知识角色、写入范围和完成条件；它也值得独立发布和回退。

按这个标准，本项目有五个业务 Harness：

| 业务 Harness（运行支架） | 它负责什么 | 完成证据 |
| --- | --- | --- |
| 选人 | 增加、删除、替换和调整队伍 | 真实页面中的当前队伍 |
| 配装 | 分析、比较、预览和应用角色配置 | 真实角色配置页 |
| 排轴 | 添加、移动、替换和应用技能按钮 | 真实时间轴 |
| BUFF | 查询、添加、替换和批量处理 BUFF | 真实按钮 BUFF 状态 |
| 计算统计 | 计算、比较、归因和解释 | 绑定方案与公式版本的结果 |

所以“当前角色”是实体，“适合当前队伍”是目标条件，武器和护甲是候选的一部分，`apply_loadout` 是 Tool；**“搭一套装备”才是一项完整业务。**

原子化保护的是完整责任和独立迭代，不是把系统切得越碎越好。

### 3、业务拆分？No，是领域问题求解 Harness（运行支架）及其可迭代体系

如果只是把代码分成五个目录，事情并没有完成。

每个业务 Harness 都要有自己的**PSM、知识角色、事务状态、工具范围、写入边界、失败恢复、完成验证和版本**。它们可以共享 Tools、知识和产品状态，但不能共享模糊的业务所有权。

跨业务请求也不需要一个“万能 Harness”。Manager（管理器）可以保存总体计划，再依次创建选人、配装、排轴、BUFF 和计算事务；每一步仍由自己的业务 Harness 负责。

最终得到的不是五份更长的 Prompt，而是**五项可以独立运行、换版、验证和追溯的领域问题求解能力**。

<details>
<summary>展开：两个容易被文件结构误导的例子</summary>

`selectedBuff` 可能保存在时间轴按钮上。排轴 Harness 负责按钮身份、位置和顺序；BUFF Harness 负责同一按钮上的 BUFF。字段住在同一个对象里，不代表两个业务可以互相修改对方的责任范围。

用户提出“换人、配装、排轴、上 BUFF，然后计算”时，Manager 保存的是跨业务计划：

```text
选人 → 配装 → 排轴 → BUFF → 计算
```

前一步失败、被拒绝或过期，后一步就不能继续消费旧结果。

</details>

## 开发与训练（未完待续）

训练改变模型本身，Harness 开发改变模型工作的环境。

Agent 做错一件事时，先别急着继续加 Prompt。先判断是模型不会推理、Tool 缺能力、知识已过期、PSM 有缺口，还是 Runtime 没守住边界。

<details>
<summary>展开：故障归因速查</summary>

| 问题 | 更可能需要修改 |
| --- | --- |
| 模型普遍无法理解语言或完成基础推理 | 模型、训练或推理能力 |
| 缺少真实读取、计算或写入能力 | Typed Tool / 产品接口 |
| 游戏事实错误或过期 | Catalog、公式或领域知识 |
| 配装常漏证据、候选不完整或不会处理修正 | 配装 Harness 版本 |
| 模型能绕过审批、越界写入或使用旧 proposal | Runtime / Product Kernel |
| 系统说完成，但真实页面没有变化 | 完成验证与 Trace |

</details>

一个更强的模型，会提高判断质量。一个更成熟的 Harness，则让判断发生在正确的事实、能力和边界里，并让结果可以验证、追溯和修改。

垂直领域 Agent 最后交付的，不是某个模型单独有多聪明，而是**模型、Harness 和真实产品共同形成的业务能力。**
