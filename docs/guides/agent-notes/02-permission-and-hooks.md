# Permission 与 Hook

工具能不能被模型看见，和工具执行时要不要征求同意，是两个不同的问题。

前者属于能力范围：这个 Agent 是否拥有读取文件、修改产品数据或执行命令的能力。后者属于 **Permission**：一次具体调用是直接允许、明确拒绝，还是停下来询问用户。

常见的权限结果可以简化成三种：

```text
allow  直接执行
deny   不允许执行
ask    等待用户确认
```

关键是，批准必须发生在模型之外。模型可以解释为什么想做这件事，却不能在自己的回答里写一句“用户已经同意”就完成自我授权。真正的 Approval 应由运行时暂停工具调用，通过界面或协议取得用户决定，再继续执行。

## Hook 不负责批准

Hook 是执行过程中的插入点。例如工具运行前补充日志、调整参数，运行后记录结果；也可以在模型请求前组装额外上下文。它更像中间件。

Permission 决定“可不可以”，Hook 处理“执行前后还要做什么”。两者可以相邻出现，但不应该混成一层。一个 PreToolUse Hook 可以发起权限检查，最终作出批准决定的仍然是权限系统。

## 修改操作最好分成两步

对不可逆或影响较大的操作，只问一句“允许吗”往往还不够。用户需要先看到模型准备改什么。于是常见做法是把执行拆成 Prepare 和 Apply：

```text
生成预览 → 检查差异与风险 → 用户批准 → 正式应用 → 验证结果
```

如果执行结果无法确认，系统不应该乐观地告诉模型“已经成功”。这种原则通常叫 Fail Closed：不确定时停住，把事实说清楚，而不是猜一个成功状态。
