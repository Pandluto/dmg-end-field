# Permission 与 Hook

读取当前配装和替换当前武器，都可以写成 Typed Tool。两者的参数可能同样规范，影响却完全不同：前者只读取事实，后者会改变用户正在使用的方案。

工具是否向模型开放，决定它“有没有这项能力”；一次具体调用是允许、拒绝还是等待确认，则属于 **Permission**。

```text
allow  直接执行
deny   明确拒绝
ask    等待用户确认
```

批准必须发生在模型之外。模型可以解释为什么想换武器，却不能在自己的回答里写一句“用户已同意”，然后完成自我授权。真正的 Approval 要由 Runtime 暂停调用，通过界面或协议取得用户决定，再继续执行。

## Hook 插在流程里，Permission 作出决定

**Hook** 是执行流程预留的插入点。工具运行前可以补充日志、调整输入或触发检查，运行后可以记录结果；模型请求开始前，也可以用 Hook 组装动态上下文。

不同 Runtime 会把 Permission 放进 Hook，或单独实现一层权限模块。名字不是重点，语义要分清：Hook 说明“在哪里介入”，Permission 说明“这次调用能不能继续”。

```text
模型请求工具
→ PreToolUse Hook
→ Permission：allow / deny / ask
→ handler 执行
→ PostToolUse Hook
```

## 高影响修改先给人看

替换武器这类操作，即使弹出一句“允许吗”，用户也未必知道模型准备改什么。更稳妥的做法是把修改拆成两步：先生成可检查的预览，再应用已经批准的版本。

DEF 将它们称为 **Prepare** 和 **Apply**：

```text
生成预览 → 检查差异与风险 → 用户批准 → 正式应用 → 核对结果
```

Apply 返回成功，也不能立刻等同于“武器已经换上”。系统还要重新读取产品状态，确认目标角色、武器和版本都符合预期；如果无法确认，就保留失败或结果未知，而不是替用户宣布成功。

安全边界到这里有了形状，但 Agent Loop 还会不断累积消息、工具结果和规则。模型下一轮究竟能看到哪些内容，记在[上下文、Skill 与 Memory](./03-context-skill-memory.md)里。
