# Permission 与 Hook

读取当前配装和修改当前配装可以是两个 Typed Tool。前一个只读，后一个会产生副作用。即使参数都合法，它们也不该得到相同的处理。

工具是否暴露给模型，属于能力范围；一次具体调用是直接允许、明确拒绝，还是停下来询问用户，属于 **Permission**。

常见的权限结果可以简化成三种：

```text
allow  直接执行
deny   不允许执行
ask    等待用户确认
```

关键是，批准必须发生在模型之外。模型可以解释为什么想做这件事，却不能在自己的回答里写一句“用户已经同意”就完成自我授权。真正的 Approval 应由运行时暂停工具调用，通过界面或协议取得用户决定，再继续执行。

## Hook 与授权不是一层语义

Hook 是执行过程中的插入点。例如工具运行前补充日志、调整参数，运行后记录结果；也可以在模型请求前组装额外上下文。不同 Runtime 可能让 Hook 返回 `allow / deny / ask`，也可能把 Permission 做成独立模块。

这里需要区分的是机制和授权语义：Hook 提供“在哪里插入检查”，Permission 说明“这次调用能否继续”。无论实现把决定放在哪个回调里，Approval 都必须来自模型之外。

一条典型链路是：

```text
模型请求工具
→ PreToolUse Hook 记录或补充调用信息
→ Permission 决定 allow / deny / ask
→ handler 执行
→ PostToolUse Hook 记录结果
```

## 修改操作最好分成两步

对不可逆或影响较大的操作，只问一句“允许吗”往往还不够。用户需要先看到模型准备改什么。通用要求是先预览、后执行；DEF 把这两步叫作 Prepare 和 Apply：

```text
生成预览 → 检查差异与风险 → 用户批准 → 正式应用 → 验证结果
```

Apply 返回以后还要检查真实产品状态。如果无法确认武器是否已经换上，系统不报告成功，而是保留“结果未知”或失败状态。安全和授权环节也采用类似的 fail-closed 姿态：检查失败时不放行。

工具调用跑得越久，Session 里的消息和结果越多。它们如何进入下一轮模型请求，记在[上下文、Skill 与 Memory](./03-context-skill-memory.md)里。
