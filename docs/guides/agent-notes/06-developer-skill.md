# 开发者自己的 Skill

有些工作做一次就结束，有些工作会反复出现：每次都要读同样的说明，检查同样的边界，按同样的顺序取证。后者才适合整理成 Skill。

Skill 不是一段更长的 Prompt。它是一份可按场景发现的工作方法，通常由入口说明和按需加载的资源组成：

```text
skill-name/
  SKILL.md
  references/
  scripts/
  assets/
```

最重要的往往不是正文写了多少，而是 `name` 和 `description` 是否准确。运行时先看到的是 Skill 目录；描述没有说清触发场景，模型就不会在正确的时候加载它。描述写得过宽，又会让无关任务反复命中。

## 正文应该写什么

一份实用的 Skill 至少要说清四件事：

1. 什么信号出现时使用它；
2. 需要按什么顺序工作；
3. 哪些动作允许，哪些动作必须停下；
4. 怎样判断结果真的完成。

长篇背景资料放进 references，确定性的重复操作尽量交给 scripts，模板和静态文件放进 assets。正文负责路由和流程，不需要把所有材料复制一遍。

Skill 也不能绕过 Runtime 的权限。它可以提醒 Agent 在修改前检查差异，却不能把原本需要 Approval 的工具变成自动允许。方法说明和能力授权属于不同层。

## 这个项目有两类 Skill

开发 Codex 使用的 Skill 放在 `.agents/skills/**`，路由集中记录在 `.agents/skill-routing.md`。它们服务于仓库开发、审计和维护。

DEF 产品运行时使用的 Skill 放在 `agent/runtime/def/skills/**`。它们面对最终用户，只能依赖产品明确开放的 Typed Tools 和权限边界。

两个目录看起来相似，身份却完全不同。开发者 Skill 可以指导 Codex 检查仓库；产品 Skill 不能借此获得任意项目读取、Shell 或开发工具。把它们混在一起，会让测试环境和产品能力边界都变得含糊。

## 最后要验证路由，而不只是验证文件

创建 `SKILL.md` 只能说明文件存在。真正需要验证的是：用户用自然语言提出任务时，模型能否发现正确 Skill；加载后是否按流程选择工具；遇到权限和失败时是否停在正确位置。

如果只有一句提醒，写进现有说明可能更合适；如果逻辑必须每次强制执行，它也许应该成为代码或 Policy，而不是 Skill。Skill 最适合承载可复用、需要判断、又不值得硬编码进 Runtime 的工作方法。

读到这里，可以回到[笔记入口](./README.md)按主题查阅。项目内实际创建或修改 Skill 时，仍应以根目录 `AGENTS.md` 和对应 Skill 的正式说明为准。
