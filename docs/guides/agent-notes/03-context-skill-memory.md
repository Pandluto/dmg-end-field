# 上下文、Skill 与 Memory

模型每次生成之前，程序都要把它需要看到的内容组装起来。这里面通常有 System Prompt、对话历史、工具定义、工具结果，以及当前任务相关的补充信息。模型本轮能看到的全部内容，就是它的上下文。

System Prompt 也不一定是一整段写死的文字。实际项目里，它往往由稳定部分和动态部分拼起来：身份、基本规则和通用工具说明尽量保持稳定；当前任务、选中对象和刚加载的知识放在后面。这样既容易维护，也更有机会利用模型服务提供的 Prompt Cache。

## Context Window 总会用完

上下文窗口有固定上限。超过上限之前，系统通常会做 Compaction：总结较早的对话，裁掉已经不重要的工具输出，同时保留最近几轮的原文。

压缩不是删除聊天记录。完整记录仍可以持久化，只是不再全部进入本轮推理。这个区别很重要：Session History 说的是“发生过什么”，Context 说的是“模型此刻看到了什么”。

Prompt Cache 和 Compaction 也不是一回事。缓存尝试复用两次请求中相同的前缀，减少重复计算；压缩是在内容太多时改变送给模型的材料。

## 不要把所有知识都塞进 System Prompt

当规则和知识越来越多，最直接的做法是继续往 System Prompt 里加。短期方便，长期会让每轮请求都携带大量无关内容。

Skill 提供了另一种组织方法：先只告诉模型有哪些 Skill，以及它们适合什么场景；真正命中任务时，再加载完整说明、引用或脚本。这就是渐进加载。

Skill、Knowledge 和 Memory 经常被混在一起，其实关注点不同：

- Skill 说明一类事情怎么做。
- Knowledge 提供完成任务需要的事实材料。
- Memory 保存跨一段时间仍值得记住的信息。

某次工具返回的内容不自动等于 Memory，一份 Skill 也不应该冒充事实数据库。把边界分清，后续才知道信息应该存在哪里、什么时候加载、过期后如何处理。

上下文管理解决了“一个 Agent 能记住多少”。任务再大一些，还会遇到“一个执行者怎么安排这么多工作”。这就到了 [Plan、Task 与 Subagent](./04-plan-task-subagent.md)。
