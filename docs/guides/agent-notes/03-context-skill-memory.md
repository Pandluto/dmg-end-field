# 上下文、Skill 与 Memory

模型准备下一步之前，需要同时看到用户目标、刚读到的配装、允许调用的工具和已经发生的对话。程序把这些材料组装成当前 Context。

System Prompt 也不一定是一整段写死的文字。实际项目里，它往往由稳定部分和动态部分拼起来：身份、基本规则和通用工具说明尽量稳定；当前任务、选中对象和刚加载的知识按本轮状态加入。

## Context Window 有上限

上下文窗口有固定上限。超过上限之前，系统通常会做 Compaction：总结较早的对话，裁掉已经不重要的工具输出，同时保留最近几轮的原文。

压缩不是删除聊天记录。完整记录仍可以持久化，只是不再全部进入本轮推理。这个区别很重要：Session History 说的是“发生过什么”，Context 说的是“模型此刻看到了什么”。

有些模型服务还提供 Prompt Cache，用来复用多次请求中相同的前缀，减少成本或延迟。它是性能优化，不负责记忆，也不替代 Compaction。

## Skill、Knowledge 和 Memory 放的不是同一种东西

规则和资料如果全部写进 System Prompt，每轮请求都会携带大量无关内容。Skill 提供了另一种组织方法：先暴露名称和描述；任务命中后，再加载完整流程、引用或脚本。这就是渐进加载。

仍以调整配装为例：

- Skill 可以记录“读取当前状态、核对攻略、生成预览、请求批准”的工作顺序。
- Knowledge 可以是武器属性、角色机制或项目目录这类可读取事实。
- Memory 可以记住用户跨会话仍然有效的偏好，例如“不要自动替换限定武器”。

某次工具返回的内容不自动等于 Memory，一份 Skill 也不应该冒充事实数据库。把边界分清，后续才知道信息应该存在哪里、什么时候加载、过期后如何处理。

Context 决定模型这一刻能看到什么；Memory 讨论哪些信息值得跨时间保留。任务本身如何拆分，则是 [Plan、Task 与 Subagent](./04-plan-task-subagent.md) 的问题。
