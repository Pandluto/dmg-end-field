# 上下文、Skill 与 Memory

模型准备下一步时，需要同时看到用户目标、刚读到的配装、可用工具和已经发生的对话。Runtime 把这些材料组装成当前 **Context**；模型只会根据这次请求里真正出现的内容继续推理。

System Prompt 也不必是一整段写死的文字。身份、基本规则和通用工具说明可以放在稳定前缀里；当前任务、选中对象和刚加载的知识，则按本轮状态接在后面。这样既方便维护，也更容易命中 Prompt Cache。

## 对话记得住，不等于本轮都看得见

Context Window 有上限。对话越来越长以后，系统会做 **Compaction**：把较早的过程压成摘要，裁掉已经没有必要反复发送的大段工具输出，同时保留最近几轮原文。

压缩的只是本轮输入，不是历史记录。完整 Session 可以继续持久化，用来回看“发生过什么”；Context 只回答“模型此刻看到了什么”。两者混在一起，就容易误以为数据库里有记录，模型便一定记得。

Prompt Cache 也不是记忆。它复用多次请求中相同的前缀，减少成本或延迟；前缀一旦变化，缓存可能失效，但 Session 和 Memory 并不会因此消失。

## Skill、Knowledge、Memory 各放什么

规则和资料如果全部塞进 System Prompt，每次请求都会携带大量无关内容。**Skill** 先暴露名称和用途，任务命中以后再加载完整流程、参考资料或脚本；这种按需展开，通常称为渐进加载。

仍以替换武器为例：

- Skill 记录“先读当前状态，再核对资料，最后生成预览”的工作方法；
- Knowledge 保存武器属性、角色机制和项目目录这类可查事实；
- Memory 记住跨会话仍然有效的偏好，例如“不要自动替换限定武器”。

某次工具结果不会自动变成 Memory，Skill 也不应该冒充事实数据库。分清这三类内容，才能知道它们从哪里来、何时加载，以及过期后应该更新哪一处。

Context 解决模型这一刻看见什么，Memory 解决什么值得跨时间保留。当任务从“换一件武器”扩大到“检查整支队伍”时，还会遇到另一个问题：工作怎样拆开，哪些部分值得交给单独的执行者。下一页继续看 [Plan、Task 与 Subagent](./04-plan-task-subagent.md)。
