# Plan、Task 与 Subagent

“换一件武器”通常不需要复杂规划。把目标扩大成“检查整支队伍的配装，找出冲突，再给出修改建议”，工作才真正出现顺序、依赖和分工。

这几个词经常一起出现，负责的事情却不同：

| 名称 | 在这次检查里负责什么 |
| --- | --- |
| Plan | 先读全队配置，再逐人检查，最后汇总建议。它描述顺序和依赖。 |
| Todo | 记录“待读取、检查中、待汇总”等进度。 |
| Task | 把“检查角色 A 的装备冲突”变成一块可以单独交付的工作。 |
| Subagent | 在独立上下文里接收这块 Task，完成检查并返回结论。 |

Plan 不是把结果提前写出来，Todo 也不是另一份 Plan。前者帮助 Agent 不忘记整体路径，后者让运行中的状态可见；Task 划出工作边界，Subagent 才是实际接手这块工作的执行者。

## 为什么要单独开上下文

检查一名角色可能要读取很多资料。如果所有过程都塞回主 Session，整队检查还没完成，上下文就已经被中间信息占满。Subagent 可以只拿到角色配置、检查规则和期望输出，做完以后把结论交回父 Agent，不必回传完整消息历史。

这种 **Context Isolation** 让主线更干净，却不应该顺带扩大权限。子 Agent 能调用哪些工具、能否继续派生子 Agent、结果按什么格式返回，都要由父任务和 Runtime 明确限制。

任务多也不等于一定要并行。几个步骤会频繁修改同一状态，或强依赖刚刚发生的对话时，拆开反而增加同步成本。先找到可以独立验收的 Task，再决定是否分派。

Plan、Task 和 Subagent 都是产品能力，不是 Agent 的必选套餐。DEF 当前收窄了这组入口，主要路径仍是单 Session 内的工具循环；它更关心工具怎样读到产品事实、修改怎样进入草稿，以及应用后如何验证。
