# 从聊天到 Agent

一次普通模型调用很简单：发出消息，拿回生成结果。回答结束，请求也结束。

模型本身不会真的打开文件、修改数据库或点击界面。它负责生成；外层程序负责保存状态、执行动作，再把结果送回模型。这里说的 **Agent**，指的是模型和这层运行机制组成的整体。

假设用户说：“读取当前配装，把角色 A 的武器换成 B。”模型第一次生成时还不知道当前配装是什么。它得先请求读取，拿到结果，再决定修改内容。单次模型生成无法完成这个往返。

## Session 先把对话接起来

**Session** 用来保存这段连续过程。里面有一条条 Message；某些 Runtime 还会把工具调用、工具结果和其他内容拆成更小的 Part 保存。

这里故意省略了消息 ID、状态字段一类实现细节，只看主要关系：

```text
Session
  ├─ 用户消息
  ├─ 助手消息
  │    └─ 工具调用
  ├─ 工具结果
  └─ 助手继续回答
```

一次用户提问通常叫一个 Turn。一个 Turn 里，模型可能不止生成一次：它先要求调用工具，拿到结果后再继续。这就需要循环。

## Agent Loop

外层程序用一个带停止条件的循环把这些步骤接起来：

```python
while True:
    response = call_model(messages, tools)

    if response.has_tool_calls:
        results = execute_tools(response.tool_calls)
        messages.extend(results)
        continue

    return response.text
```

循环保存消息、执行工具、把结果放回上下文，并判断什么时候该停。只要模型还在请求工具，循环就继续；当它给出最终回答，或者遇到错误、取消、步骤上限，循环才结束。

在这个例子里，读取配装和准备修改可能对应两个不同工具。模型怎样准确表达要调用哪个函数、传什么参数，记在 [Tool Use](./01-tool-use.md) 里。
