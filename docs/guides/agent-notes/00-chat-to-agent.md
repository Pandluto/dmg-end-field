# 从聊天到 Agent

一次普通模型调用只有一来一回：用户发出消息，模型生成回答，这次请求就结束了。它可以告诉你“应该换哪件武器”，却不会因此真的读到当前配装，更不会自己改掉产品数据。

模型负责生成下一步，外层程序负责保存对话、执行动作，再把动作结果送回模型。这里所说的 **Agent**，就是模型和这套运行机制组成的整体。

## 为什么一次生成不够

假设用户说：“读取当前配装，把角色 A 的武器换成 B。”模型起初并不知道角色 A 正在用什么。它要先请求读取，看到结果后才能准备修改；修改完成以后，还要再看一次真实状态，才能确认任务结束。

这已经不是一次问答，而是一段连续过程：

```text
提出目标 → 读取状态 → 准备修改 → 执行修改 → 核对结果
```

**Session** 先把这段过程接起来。它保存用户消息、模型回复、工具调用和工具结果；一次用户输入到最终回复，通常称为一个 Turn。

一个 Turn 里，模型可能生成很多次。第一次要求读配装，第二次要求准备修改，最后一次才回答用户。外层程序需要反复判断“模型还要不要调用工具”，这就是 **Agent Loop**。

```python
while True:
    response = call_model(messages, tools)

    if response.has_tool_calls:
        results = execute_tools(response.tool_calls)
        messages.extend(results)
        continue

    return response.text
```

这段伪代码省略了权限、取消、重试和步骤上限，却保留了最重要的骨架：模型决定下一步，程序执行，结果重新进入上下文；直到模型给出最终回答，循环才停。

真正开发时，难点很快会从“怎样循环”变成“模型怎样准确表达下一步”。读取配装不能只靠一句含糊的自然语言，它需要一个程序能够验证和执行的调用，这就来到下一页：[Tool Use](./01-tool-use.md)。
