# 从聊天到 Agent

最开始的模型很像一问一答。你发一段文字，它根据已经看到的内容继续生成文字。回答结束，这一轮也就结束了。

这件事容易被“智能”两个字说得太玄。单看模型，它并不会真的打开文件、修改数据库或者点击界面。它只是生成下一段最合适的内容。让模型能够连续观察、决定、行动的，是外面那层程序。我们通常把模型和这层运行机制合在一起叫作 **Agent**。

## Session 先把对话接起来

如果每次请求都互不相干，模型不会知道上一轮发生了什么。于是需要 **Session** 保存连续对话。Session 里面有一条条 Message；Message 除了用户和助手的文字，还可能包含推理片段、工具调用和工具结果。

可以先用一个不太严格、但够用的关系理解它们：

```text
Session
  ├─ 用户消息
  ├─ 助手消息
  │    └─ 工具调用
  ├─ 工具结果
  └─ 助手继续回答
```

一次用户提问通常叫一个 Turn。一个 Turn 里，模型可能不止生成一次：它先要求调用工具，拿到结果后再继续。这就需要循环。

## Agent Loop 是真正的骨架

Agent Loop 没有什么神秘的，本质上就是一个带停止条件的循环：

```python
while True:
    response = call_model(messages, tools)

    if response.has_tool_calls:
        results = execute_tools(response.tool_calls)
        messages.extend(results)
        continue

    return response.text
```

模型仍然只负责生成。循环负责保存消息、执行工具、把结果放回上下文，并判断什么时候该停。只要模型还在请求工具，循环就继续；当它给出最终回答，或者遇到错误、取消、步骤上限，循环才结束。

所以 Agent 和普通聊天最关键的区别，不是模型突然有了另一种智力，而是它获得了一个可以反复“看结果再行动”的运行环境。

接下来的问题也很自然：模型说“我要读取文件”之后，程序怎么知道它想调用哪个函数、参数又是什么？这就是 [Tool Use](./01-tool-use.md) 要解决的事。
