# 修改大纲—交互优化

## [阶段目标]

本阶段目标是将当前前端交互层从“组件视图切换”升级为“先进工作台 + 页面化”的产品结构。

当前项目已经存在 `selection` 和 `canvas` 两个视图语义，但它们仍由 `AppContext.currentView` 直接控制。后续应将其升级为真实页面结构：

- `/select`：选人页。
- `/timeline`：排轴工作台页。

本大纲只规划 UI / 交互架构，不混入战斗机制、增幅区、Buff 缓存主链路或伤害公式修改。

本阶段优先交互先行，不启动整体视觉重设计，不要求一次性替换全部 `CanvasBoard` 布局。

## [当前交互现状]

### 1. 页面语义已存在，但不是正式页面

当前 `src/App.tsx` 根据 `state.currentView` 分发：

- `selection` 渲染 `SelectionPanel`。
- `canvas` 渲染 `CanvasBoard`。

问题：

- 页面边界不清晰。
- 浏览器地址无法表达当前页面。
- 刷新、返回、直接访问目标页面缺少路由语义。
- 后续扩展工作台页面、配置页面、详情页面时会继续堆在 view state 上。

### 2. CanvasBoard 承担过多工作台职责

当前 `CanvasBoard` 同时承担：

- 排轴画布容器。
- 侧边功能面板布局。
- 技能沙盘。
- 干员配置弹窗。
- timeline 恢复。
- skill button 恢复。
- Buff 事件监听。
- 按钮拖拽协调。
- 右键删除和锁定逻辑。
- 保存入口。

问题：

- `CanvasBoard` 更像整个工作台页面，而不是单一画布组件。
- 后续继续扩功能会使 `CanvasBoard` 继续膨胀。
- 顶部栏、侧栏、浮层、反馈层缺少统一工作台壳。

### 3. SidePanel 还是功能标签容器

当前 `SidePanel` 是固定侧栏 + tab 切换。

问题：

- 还不是工作台工具面板体系。
- Buff、伤害、筛选、配置、详情等能力没有统一面板规范。
- 后续新增工具容易继续堆在 SidePanel 内。

### 4. 交互反馈缺少统一规范

当前反馈分散在：

- `console.log`
- `alert`
- 弹窗状态
- 局部按钮 disabled
- 拖拽视觉层

问题：

- 保存成功、拖拽失败、未选中技能按钮、Buff 已存在等反馈没有统一出口。
- 用户很难判断操作是否真正生效。

## [目标方向：先进工作台]

目标不是简单把组件换个目录，而是建立类似工作台产品的交互结构：

1. 页面清晰
   - 选人页负责“准备队伍”。
   - 排轴工作台页负责“编辑战斗时间轴”。

2. 区域清晰
   - 顶部栏负责全局操作和状态。
   - 左侧工具面板负责工具能力。
   - 中央画布负责主编辑区。
   - 右侧或浮层负责详情和配置。
   - 反馈层负责提示、状态和错误。

3. 职责清晰
   - page 层组合页面。
   - workspace shell 组合工作台布局。
   - feature panel 承载具体工具。
   - hooks 管理交互状态。
   - service / repository / calculator 继续承载业务、数据和计算。

4. 状态清晰
   - 页面路径表达当前工作区。
   - sessionStorage 继续负责业务数据恢复。
   - AppContext 只保留当前仍必要的全局 UI 状态，后续逐步收敛。

## [页面化设计]

### 1. 引入 React Router

后续实现允许新增依赖：

```bash
npm install react-router-dom
```

页面路径固定为：

- `/select`
- `/timeline`

默认重定向规则：

- `/` 默认跳转到 `/select`。
- 未匹配路径默认跳转到 `/select`。

### 2. SelectionPage

建议文件：

- `src/pages/SelectionPage.tsx`

职责：

- 渲染现有 `SelectionPanel`。
- 承接选人页级别布局。
- 提供进入 `/timeline` 的页面导航。

约束：

- 不改变选人逻辑。
- 不改变最多 4 人选择限制。
- 不在本页处理排轴、Buff、伤害计算。

### 3. TimelineWorkbenchPage

建议文件：

- `src/pages/TimelineWorkbenchPage.tsx`

职责：

- 渲染排轴工作台。
- 承接后续 `WorkspaceShell`。
- 处理进入页面时的状态检查。

推荐策略：

- 如果没有已选干员，进入 `/timeline` 时跳回 `/select`。
- 如果已有已选干员，则正常渲染工作台。

约束：

- 不改变 timeline、skill-button、buff-list 的恢复规则。
- 不改变 0.5.1 / 0.5.2 / 0.5.3 已完成的分层成果。

### 4. App 层职责

后续 `App.tsx` 只负责：

- Router 容器。
- 页面路由分发。
- 全局 provider 包裹。

不再直接根据 `currentView` 渲染 `SelectionPanel` / `CanvasBoard`。

### 5. currentView 处理策略

短期：

- 可以保留 `AppContext.currentView`，避免一次性清理所有旧入口。
- 但页面跳转应以 React Router 为准。

中期：

- 清理 `SET_VIEW` 对 UI 路由的直接控制。
- 将“返回选人”“进入排轴”改为 router navigation。

禁止：

- 不要同时让 `currentView` 和 router 互相写入，避免双真相。

## [工作台布局设计]

排轴页目标结构：

```text
TimelineWorkbenchPage
  WorkspaceShell
    WorkbenchTopBar
    WorkbenchSidePanel
    WorkbenchCanvasArea
    WorkbenchInspector / FloatingPanel
    WorkbenchFeedbackLayer
```

### 1. 顶部工作台栏

职责：

- 返回选人页。
- 显示当前队伍信息。
- 显示保存状态。
- 提供保存、恢复、清空等全局操作入口。

第一阶段不要求立即实现完整顶部栏，但大纲必须明确该方向。

### 2. 左侧工具面板

职责：

- 承载 Buff 工具。
- 承载伤害加成工具。
- 承载筛选、搜索、配置等后续工具。

现有 `SidePanel` 后续应逐步升级为工作台工具面板，而不是无限追加 tab。

### 3. 中央排轴画布

职责：

- 渲染 timeline。
- 渲染节点。
- 渲染技能按钮。
- 接收技能按钮拖放。

现有 `CanvasArea` 应保持为主画布区域，不承担页面级布局职责。

### 4. 右侧或浮层检查器

职责：

- 干员配置。
- 技能按钮详情。
- Buff 详情。
- 后续公式/伤害详情。

现有 `OperatorConfigPanel` 可以先保留为浮层，后续再纳入检查器体系。

### 5. 操作反馈层

职责：

- 保存成功。
- 拖拽失败。
- 未选择技能按钮。
- Buff 已存在。
- 缓存恢复结果。
- 错误提示。

禁止继续依赖 `alert` 作为主要反馈方式。

## [交互层拆分]

### 1. Page 层

职责：

- 页面入口。
- 页面级导航。
- 页面级布局组合。

不负责：

- 业务写入顺序。
- Buff 主从关系。
- 伤害公式。

### 2. Workspace Shell 层

职责：

- 组织顶部栏、侧栏、主画布、检查器、反馈层。
- 提供统一工作台布局。

不负责：

- 具体 Buff 添加逻辑。
- 具体伤害计算。
- 具体 storage 读写。

### 3. Feature Panel 层

职责：

- Buff 面板。
- 伤害面板。
- 配置面板。
- 后续工具面板。

要求：

- 面板只调用 hooks / service facade。
- 面板不直接写 storage。
- 面板不直接维护跨表清理规则。

### 4. Hook 层

职责：

- 页面内交互状态。
- 拖拽状态。
- 搜索状态。
- 弹窗状态。
- 局部 UI 状态。

要求：

- hooks 不直接维护跨表业务规则。
- hooks 不直接写主缓存表。
- hooks 调用 service / repository / events。

### 5. Core 层

继续沿用 0.5 已完成成果：

- `src/core/repositories`
- `src/core/services`
- `src/core/calculators`
- `src/core/events`
- `src/core/domain`

要求：

- 数据读写继续走 repository / service。
- 伤害计算继续走 calculators。
- Buff 交互继续走 `useCandidateBuffs` / `useBuffInteraction`。
- 不恢复组件直接写 storage。
