# 面板计算

> 本文档定义“面板计算”在当前项目中的方法规格、数据依赖、调用链与输出约束。  
> 它服务于后续可被前端直接调用的面板计算方法，而不是旧 CLI 的操作说明。

## 1. 文档定位

当前项目中的“面板计算”需要统一三套事实来源：

- 规则文档与基础口径；
- 参考计算器与历史实现；
- 当前前端配置界面的真实交互链路。

本文件正文以“前端可调用的方法规格”为核心，目标是让后续执行者能够直接实现：

- 从配置界面读取角色 / 武器 / 装备设置；
- 结合 `public/data` 下的角色 `max.json`、武器 `max.json`、武器 `buff.json`；
- 产出可供【配置界面——信息模块】直接渲染的文字结果与结构化面板结果。

旧 CLI 与参考实现不会删除，但只保留在附录中作为历史参考。

## 2. 目标与职责

### 2.1 方法目标

面板计算方法的目标不是单纯返回一个攻击力数值，而是同时完成三件事：

1. 基于当前前端配置，计算角色最终面板；
2. 给出角色、武器、潜能、珠子、装备等来源拆解；
3. 将结果转化为【信息模块】可直接渲染的文字结构。

### 2.2 方法职责边界

应负责：

- 读取并整合角色 `max.json` 与武器 `max.json`；
- 读取武器 `buff.json` 中会影响面板展示的口径；
- 按角色、武器、潜能、珠子、装备配置计算最终面板；
- 将计算结果拆解为“角色贡献 / 武器贡献 / 装备贡献 / 潜能贡献 / 条件说明”。

不应负责：

- 排轴逻辑；
- 技能伤害结算；
- 拖拽、画布布局或配置面板 UI 行为本身；
- 未被触发的条件效果直接并入最终面板。

## 5. 方法规格

### 5.1 输入规格

建议将面板计算实现为可直接被前端调用的方法。  
输入规格必须分清“当前前端已提供的输入”和“方法层预留输入”。

#### 当前前端已提供的输入

- `characterName`
- `weaponName`
- `weaponPotentialMode`

当前武器展示链路还隐含固定口径：

- 角色等级展示为 `90`
- 武器等级展示为 `90`
- 武器 skill1 / skill2 展示为 `9`
- 武器 skill3 展示随潜能切换为 `4 / 9`

这些属于当前 UI 展示事实，不等于完整面板方法的最终输入设计。

#### 方法层预留输入

以下字段虽尚未有完整前端输入控件，但方法层必须预留：

- `characterLevel`
- `characterPotential`
- `weaponLevel`
- `equipment`

统一输入草案如下：

```ts
interface PanelCalculationInput {
  characterName: string
  characterLevel: number
  characterPotential?: number
  weaponName: string
  weaponLevel: number
  weaponPotentialMode: 'P0' | 'PMAX'
  equipment?: EquipmentPanelInput
}
```

其中需要特别区分：

- `characterName`、`weaponName` 由配置界面选择；
- `weaponPotentialMode` 当前前端已真实存在，必须显式支持 `P0` 与 `PMAX`；
- `characterLevel`、`weaponLevel` 当前可以默认 `90`，但方法不应把它们写死成不可配置常量；
- 武器技能档位不再依赖独立 `matrixCode` 输入，而是直接读取武器模块滑块所表达的现成值；
- `equipment` 当前前端未接入，属于方法层预留输入。

其中主副能力相关字段统一改为：

| 字段 | 说明 |
|------|------|
| `mainStatBoost` | 主能力提升 |
| `subStatBoost` | 副能力提升 |
| `allStatBoost` | 全能力提升 |

### 5.2 装备输入规格

装备输入代表“其他装备加成”，不含武器贡献。

```ts
interface EquipmentPanelInput {
  mainStat?: number

  intelligence?: number

  flatAtk?: number
  atkPercent?: number
  mainStatBoost?: number
  subStatBoost?: number

  critRate?: number

  physicalDmgBonus?: number
  fireDmgBonus?: number
  electricDmgBonus?: number
  iceDmgBonus?: number
  etherDmgBonus?: number
  skillDmgBonus?: number
  chainSkillDmgBonus?: number
  ultimateDmgBonus?: number
  allSkillDmgBonus?: number
  physicalRes?: number
  fireRes?: number
  electricRes?: number
  iceRes?: number
  etherRes?: number
  voidRes?: number
  healingBonus?: number
  incomingHealingBonus?: number
  chainSkillCdr?: number
  ultimateChargeEfficiency?: number
  imbalanceEfficiency?: number
}
```

### 5.3 输出规格

输出必须同时服务“数值计算”和“信息模块渲染”。

```ts
interface PanelCalculationResult {
  summary: PanelSummary
  breakdown: PanelBreakdown
  infoBlocks: InfoModuleBlock[]
  textLines: string[]
}
```

要求：

- `summary` 用于后续伤害计算复用；
- `breakdown` 用于展示每个来源的贡献；
- `infoBlocks` 或 `textLines` 用于配置界面信息模块直接渲染；
- 输出必须保留“武器条件说明”，即使该条件不计入最终面板。

## 6. 当前口径下的关键规则

### 6.1 武器潜能与技能档位

本节必须区分“当前前端展示口径”和“后续统一方法口径”。

#### 当前前端展示口径

当前 [OperatorConfigPanel.tsx](file:///c:/Users/zsk86/Documents/trae_projects/ddd/src/components/CanvasBoard/components/OperatorConfigPanel.tsx) 中的武器展示规则为：

- `skill1 = 9`
- `skill2 = 9`
- `skill3 = 4 / 9`

展开后即：

| 潜能状态 | skill1 | skill2 | skill3 |
|------|------|------|------|
| `P0` | 9 | 9 | 4 |
| `PMAX` | 9 | 9 | 9 |

这说明当前前端武器展示呈现为 `994 / 999`，但这仍属于展示实现，不等同于一个已经完成的通用珠子/阶段解析器。

#### 后续统一方法口径

后续真正实现面板计算方法时，武器潜能规则仍按：

- `0潜`：读取 `994`
- `满潜`：读取 `999`

但应由统一方法直接读取武器模块滑块所表达的现成值，而不是再设计一套独立珠子输入。

### 6.2 装备加成不含武器

装备输入代表“其他装备加成”，不包含武器带来的属性与效果。

正确理解：

```text
角色基础 + 好感 + 武器 + 潜能 + 装备
```

错误做法：

- 先从角色主副能力中扣除武器贡献，再传入 `equipment`
- 将武器 skill1 / skill2 / skill3 的数值重复写进装备输入

### 6.3 武器参与面板的范围

面板计算只涉及武器的以下部分：

- 武器基础攻击；
- `skill1`；
- `skill2`；
- `skill3` 的无条件部分（`passive` 字段）。

不涉及：

- `skill3` 的条件触发部分；
- `buff.json` 中仅在触发后生效的武器效果；
- 任何未被明确标记为无条件生效的武器描述文本。

### 6.4 技能3的 passive 与条件效果必须区分

武器 skill3 需要分成两类：

- `passive`：无条件生效，可直接进入面板；
- 条件效果：只在满足条件时生效，不应直接并入默认面板。

因此需要明确三层口径：

- 当前现状：前端配置面板只读取武器 `max.json`，未读取 `buff.json`；
- 面板直算：`max.json` 中 `skill3.levels.*.passive` 属于默认面板口径；
- 目标方案：`buff.json` 中的条件触发效果用于触发后计算、条件说明与信息模块补充，不应默认写死到基础面板里。

### 6.5 主副能力换算与好感度

基础口径：

```text
能力值加成 = 主能力 × 0.5% + 副能力 × 0.2%
```

角色好感度加成固定写死为：

```text
主能力 + 60
```

即主能力原值应按以下来源拆解：

```text
主能力原值 = 角色主能力 + 好感60 + 武器主能力 + 潜能主能力 + 装备主能力
```

攻击力公式：

```text
攻击力 = [(角色基础攻击 + 武器基础攻击) × (1 + 百分比加成) + 固定攻击] × (1 + 能力值加成)
```

如果存在主副能力提升类加成，则按以下顺序处理：

```text
主能力' = 主能力 × (1 + mainStatBoost)
副能力' = 副能力 × (1 + subStatBoost)
若存在 allStatBoost，则主能力' 与副能力' 均继续乘以 (1 + allStatBoost)
能力值加成 = 主能力' × 0.5% + 副能力' × 0.2%
```

### 6.6 角色潜能

角色潜能包含两层规则：

#### 默认潜能口径

- 六星角色默认 `0潜`
- 其余角色默认满潜

该规则即使当前没有单独的角色潜能交互控件，也应在方法层中作为默认值实现。

#### 面板生效口径

角色潜能只在具有 `stats` 字段时才直接参与面板。

- 有 `stats`：并入面板；
- 无 `stats`：仅保留描述，不直接进入默认面板。

此口径应与 [import-character.md](file:///c:/Users/zsk86/Documents/trae_projects/ddd/docs/reference-skills/Rules/import-character.md) 保持一致。

## 7. 信息模块输出约束

当前信息模块尚未接入面板结果，因此本节描述的是**目标输出约束**。

面板计算方法调用完成后，至少应能为信息模块输出以下内容：

- 角色名、等级、潜能状态；
- 武器名、武器等级、武器潜能状态；
- 四维能力值总览；
- 主副能力来源拆解；
- 基础攻击、攻击百分比、固定攻击、能力值加成；
- 生命、攻击、暴击、源石技艺强度；
- 治疗效率、受治疗效率、充能效率、冷却缩减、失衡效率；
- 物理/元素/技能类伤害加成；
- 武器条件效果说明；
- 装备模块带来的数值与条件描述。

建议输出结构：

```text
角色面板
武器信息
能力值
装备信息
主副能力换算
基础属性
效率与抗性
伤害加成
条件说明
```

其中“条件说明”必须保留来源，例如：

- 来自武器 buff；
- 来自装备套装；
- 来自角色潜能的条件效果；
- 当前是否为默认未触发状态。

## 8. 与前端模块的关系

### 8.1 武器模块

当前武器模块已真实采集或展示：

- 武器名；
- 武器潜能；
- 武器数据展示所需的技能档位信息。

当前尚未采集：

- 珠子码。

因此本节应理解为：

- 已采集输入应逐步回收进统一方法；
- 未采集输入应先作为后续预留字段存在。

### 8.2 装备模块

装备模块当前尚未开发，但方法规格必须提前预留：

- 固定属性；
- 百分比属性；
- 效率类属性；
- 条件触发效果；
- 套装效果。

装备模块的职责与前一版本相似，只是最终需要通过统一面板方法转成信息模块文本。

### 8.3 信息模块

信息模块不应自行推导数值，只负责渲染面板方法提供的结果。

推荐原则：

- 计算逻辑在方法层；
- 显示逻辑在信息模块；
- 模块之间只通过明确的数据结构通信。

## 9. 实现约束

后续真正实现面板计算方法时，应遵守以下约束：

- 面板核心数值数据源以 `public/data/**/*max.json` 为准；
- 条件效果与补充说明数据源以 `public/data/**/*buff.json` 为准，但当前前端尚未接入；
- 不把参考代码目录误当成现网调用入口；
- 不把 CLI 参数结构直接照搬为前端唯一输入方案；
- 不把条件效果默认写进无条件面板；
- 不把装备模块未开发当成可以省略接口设计的理由；
- 方法输出必须适配信息模块文字渲染，而不仅是数学结果。
