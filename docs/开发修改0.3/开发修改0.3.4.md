## 2. 本次面板计算口径（0.3.4 唯一准则）

### 2.1 攻击力公式

```text
攻击力 = [(角色基础攻击 + 武器基础攻击) × (1 + 百分比加成) + 固定攻击] × (1 + 能力值加成)
```

### 2.2 主副能力换算

```text
能力值加成 = 主能力 × 0.5% + 副能力 × 0.2%
```

若存在主副能力提升类加成：

```text
主能力' = 主能力 × (1 + mainStatBoost)
副能力' = 副能力 × (1 + subStatBoost)
若存在 allStatBoost，则主能力' 与副能力' 均继续乘以 (1 + allStatBoost)
能力值加成 = 主能力' × 0.5% + 副能力' × 0.2%
```

### 2.3 主副能力提升字段

后续实现时统一使用以下字段命名：

| 字段 | 说明 |
|------|------|
| `mainStatBoost` | 主能力提升 |
| `subStatBoost` | 副能力提升 |
| `allStatBoost` | 全能力提升 |

### 2.4 好感度规则

- 角色好感度加成直接写死：**主能力 +60**
- 不额外设计好感输入控件
- 后续实现中必须在注释中明确：这是当前项目固定口径，不是用户可配置项

### 2.5 武器参与面板的范围

面板计算只涉及武器以下部分：

- 武器基础攻击；
- `skill1`；
- `skill2`；
- `skill3` 的无条件部分 `passive`

不进入默认面板：

- `skill3` 条件触发部分；
- `buff.json` 中仅在触发后生效的效果；
- 任何未标记为无条件生效的描述型文本。

### 2.6 武器潜能与技能档位

后续实现直接复用当前武器模块滑块的现成值，不再设计独立珠子输入。

唯一口径：

| 武器滑块 | 读取档位 |
|------|------|
| `P0` | `994` |
| `PMAX` | `999` |

展开即：

| 潜能状态 | skill1 | skill2 | skill3 |
|------|------|------|------|
| `P0` | 9 | 9 | 4 |
| `PMAX` | 9 | 9 | 9 |

### 2.7 角色潜能默认规则

- 六星角色默认 `0潜`
- 其余角色默认满潜

同时遵守：

- 只有带 `stats` 的潜能才直接进入默认面板；
- 纯描述型潜能只保留说明，不直接并入默认面板。

### 2.8 装备输入规则

- 装备输入代表“其他装备加成”
- **不包含武器贡献**
- 装备模块虽然当前未开发，但 0.3.4 的方法接口必须预留其输入位置

### 2.9 角色配置 json 规则

后续实现时，浏览器本地应为每个角色维护一份配置 json。

建议结构：

```json
{
  "characterId": "角色唯一标识",
  "characterName": "角色名",
  "characterPotential": 0,
  "weaponName": "武器名",
  "weaponPotentialMode": "P0",
  "equipment": {},
  "panelSnapshot": {},
  "infoSnapshot": []
}
```

要求：

- `weaponName`、`weaponPotentialMode` 必须与配置面板联动；
- `characterPotential` 当前可按默认规则自动填充；
- `equipment` 当前先保留空结构；
- `panelSnapshot` 用于缓存上一次面板计算结果；
- `infoSnapshot` 用于缓存信息模块可直接渲染的数据；
- 存储层采用“运行时内存态 map”，以满足“刷新/重启即清空”的要求。

## 4. 技术方案设计

## 4.1 方法层

- 新增统一面板计算方法，负责整合角色、武器、潜能、装备输入。
- 方法输出不应只返回一个 `ATK`，而应返回：
  - 面板汇总值；
  - 来源拆解；
  - 信息模块可直接消费的文字块或文本行。
- 方法内部必须显式区分：
  - 当前默认面板；
  - 条件说明；
  - 未来可触发效果。
- 方法调用时优先读取“当前角色配置 json”中的配置值，而不是直接从零散组件状态拼接最终输入。

建议方法拆分（实现导向，贴近当前组件）：

- `buildCharacterConfigKey(characterId)`：统一角色配置索引键；
- `initCharacterConfig(character)`：初始化角色配置 json；
- `mergeCharacterConfig(prev, patch)`：局部更新角色配置；
- `calculatePanelFromCharacterConfig(config, characterMax, weaponMax, weaponBuff)`：核心计算；
- `buildInfoSnapshot(panelResult)`：构建信息模块渲染数据。

## 4.2 本地角色配置存储层

- 在角色选择完成后，为每个角色初始化一份浏览器本地角色配置 json。
- 存储以 `OperatorConfigPanel` 内存态为主（`characterConfigMap`），不引入持久化存储。
- 每个角色配置 json 按角色唯一标识建立索引，避免角色切换时串数据。
- 配置面板的武器选择、武器潜能切换后，立即回写对应角色配置 json。
- 信息模块读取角色配置 json，而不是直接读取零散控件状态。

vibe coding 落地约束：

- 先复用现有 `weaponNameMap` / `weaponPotentialModeMap`，再收敛为统一 `characterConfigMap`；
- 任何配置写回失败都不能阻断面板渲染（计算失败降级为占位文本）；
- 清理策略可测试：刷新/重启后配置回到默认初始值。

## 4.3 数据层

角色数据来源：

```text
public/data/characters/{角色名}/{角色名}max.json
```

武器数据来源：

```text
public/data/weapons/{武器名}/{武器名}max.json
```

武器条件说明来源：

```text
public/data/weapons/{武器名}/{武器名}buff.json
```

使用原则：

- `max.json` 负责默认面板与无条件面板部分；
- `buff.json` 负责条件说明与触发后扩展效果；
- 当前阶段禁止把 `buff.json` 条件效果默认写进基础面板。

防幻觉约束（数据层）：

- 不推测字段：只读取文件中存在且有规范定义的字段；
- 不发明映射：若字段缺失，保留空值并写明“未接入/无数据”；
- 不跨层兜底：`buff.json` 不反向覆盖 `max.json` 的无条件面板值。

const [characterConfigMap, setCharacterConfigMap] = useState<Record<string, CharacterConfig>>({})

const characterStateKey = resolvedActiveCharacterId ?? '__panel__'

useEffect(() => {
  if (!resolvedActiveCharacterId || !activeCharacter) return
  setCharacterConfigMap(prev => {
    if (prev[characterStateKey]) return prev
    return {
      ...prev,
      [characterStateKey]: initCharacterConfig(activeCharacter)
    }

const handleWeaponSelect = (weaponName: string) => {
  // 保留现有 weaponNameMap 写法

  // 新增：同步写回角色配置

}

const handleWeaponPotentialToggle = () => {

  // 保留现有 weaponPotentialModeMap 写法

}

## 4.5 信息模块渲染层

信息模块后续只负责渲染，不自行推导数值。

应展示：

- 角色名、等级、潜能状态；
- 武器名、武器等级、武器潜能状态；
- 四维；
- 主副能力换算；
- 基础攻击与攻击力；
- 暴击、源石技艺强度、效率类属性；
- 武器条件说明；
- 装备说明。

读取原则：

- 信息模块优先读取当前角色配置 json 中的 `infoSnapshot`；
- 若 `infoSnapshot` 不存在，则读取最新面板计算结果并回写；
- 禁止信息模块直接从武器模块滑块或其他零散控件状态自行推导文本。

vibe coding 细则（渲染层）：

- 先做“最小文本块渲染”（角色名/武器名/攻击力）；
- 再逐步接入四维、主副能力拆解、效率、条件说明；
- 每次只新增一组字段并验证，避免一次性接满导致定位困难。

## 4.6 注释与可读性标准（强制）

- 面板计算入口必须写注释，说明输入从哪里来、输出给谁用。
- 主副能力换算必须写注释，说明先后顺序与乘区边界。
- 武器 `passive` 与 `buff.json` 区分必须写注释，避免后续把条件效果错误并入默认面板。
- 默认潜能规则必须写注释，明确“六星 0 潜，其余满潜”为当前项目口径。
- 信息模块渲染层必须写注释，明确“只渲染，不推导”。

- 将面板结果与信息结果回写到角色配置 json。
- 完成最小闭环：条件说明可见、默认面板不受条件触发影响。

### 阶段 D：信息模块接入

- 配置界面调用统一面板方法；
- 信息模块从角色配置 json 读取输出结果并进行文字渲染；
- 保证后续伤害计算能够复用面板结果结构。
- 完成最小闭环：切换角色后信息模块自动切换到该角色的配置快照。

### 阶段 E：回归与验收

- 检查六星 / 非六星默认潜能是否正确；
- 检查 `P0 / PMAX` 是否正确映射为 `994 / 999`；
- 检查武器条件效果未误算进默认面板；
- 检查装备接口已预留但未污染当前武器链路；
- 检查浏览器刷新或重启后，本地角色配置已自动清空（回到默认值而非脏数据）。
- 检查任何一步失败时可快速回退到上一阶段稳定态。
