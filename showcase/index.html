SYSTEM ARCHITECTURE / README-ALIGNED

Electron 把方案留在本机；
OpenCode 让 AI 在边界里干活。

不用背架构图：方案先存好，AI 再拿指定工具试做；它的改动先出现在草稿里。想换一套 AI 能力？交给 Harness 在新会话里考试，通过后才上岗。

01 / ELECTRON → SQLITE你做的方案，先变成可恢复的本地档案
React Workbench你在这里选人、配装、排轴、计算
→
Electron Bridge只在你的电脑上把数据送到下一站
→
SQLite Repository可恢复的方案档案，存在本机 AppData

每一份方案都能保存快照、保留 AI 草稿、记录当前版本与操作历史；分享包会先校验，再一次性写入，避免半截数据覆盖你的方案。

02 / OPENCODE → WORK NODE它能帮你试，但不能偷偷替你改
OpenCode Runtime负责听懂你的问题、调度本地 AI 会话
→
DEF Adapter先给 AI 划好终末地任务与能力边界
→
Typed Tools只准碰角色、装备、伤害和排轴这些指定操作
→
Work Node先在草稿里试，告诉你 Diff、风险与校验结果

它没有任意 Shell 或文件权限。真正的修改会先变成草稿，展示改了什么、有什么风险；你可以检查、应用，也可以直接丢掉。

03 / HARNESS HOT-SWAP热插拔：新对话试新能力，旧对话不受影响
发现一次真实失败导出、脱敏、定位原因
→
Harness Package把候选“脑回路”做成可比较、带 hash 的版本包
→
Session Pinning只让新对话绑定候选 hash；旧对话保持原样
→
拿真实场景回放考试修复 · 回归 · 安全三项检查
→
人来决定是否上岗stable / candidate / rollback
热插拔 到底是什么？不是聊到一半给 AI 换人格。每个新 session 创建时绑定自己的 DefHarnessSessionBindingV1 与内容 hash；通过或回滚只影响之后的新对话，所以可以比较，也能安全回退。旧对话保持原样
04 / SQLITE NODE TREE + GITHUB RELEASE攻略方案与图片资源，各走各的路

你的攻略方案 / 本机事实源

一份排轴方案队伍 · 配装 · 按键 · Buff
→
SQLite内容 hash、原子导入与本机保存
→
Work Node Tree把快照、草稿、Checkout 与审计串起来

节点树不是装饰图：它把保存、恢复、AI 草稿、校验、审批与回退串起来，所以你随时知道这套攻略从哪里来、改过什么。

GitHub Release / 图片资源更新

GitHub Release→版本检查与下载→本地图片资源

立绘等图片走发布更新；你的排轴和攻略方案仍只在本机 SQLite 里保存。更新图片，不会覆盖你的方案。

你的方案在哪？在本机 SQLite，不在浏览器缓存，也不依赖云端同步。

AI 为什么不会乱来？它没有任意 Shell 和文件权限，只能调用规定好的领域工具。

更新 AI 凭什么可信？候选能力要在真实会话中回放，通过修复、回归与安全检查后才由人批准。
